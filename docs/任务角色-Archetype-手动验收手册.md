# 任务角色 Archetype 手动验收手册

## 文档说明

本文档只保留**不能稳定自动化验证**、需要你人工观察和判断的验收项。

以下内容**不在本文档中**，因为我已经可以自动化验证或它们更适合自动化验证：
- 前端构建是否通过
- Rust 后端编译是否通过
- Tauri 打包是否通过
- 应用是否可以启动
- 本地 archetype 目录和 manifest 是否生成
- system archetype 文件是否已同步
- archetype 同步单元测试是否通过

---

## 一、手动验收范围

本轮需要你人工验收的内容主要是这 5 类：

1. Task Builder 和 Add Role 弹窗的交互体验是否正常
2. archetype 选择后的说明信息是否正确、可理解
3. 多角色任务的顺序、展示和切换是否符合预期
4. `system_prompt_override` 是否对角色行为产生可见影响
5. 手动 forward / handoff 相关交互是否仍然顺畅

---

## 二、验收前准备

请先启动应用，并进入可正常创建任务的主界面。

建议本轮至少准备：
- 2 个可用 provider/model 组合（如果当前系统支持）
- 一个你熟悉的代码目录作为工作目录
- 一个方便你观察差异的简单任务主题，例如：
  - “为当前项目写一个功能开发计划”
  - “分析当前页面组件结构并提出改进建议”

---

## 三、手动验收步骤

## 1. 验证 Task Builder 基础交互

### 操作步骤
1. 打开 `New Task` / 新建任务弹窗。
2. 观察弹窗整体布局。
3. 不添加角色，直接查看初始状态。
4. 尝试点击关闭按钮、遮罩层关闭、取消按钮。

### 需要观察的内容
- 弹窗是否能稳定打开、关闭
- 页面是否有明显样式错乱、按钮遮挡、文字重叠
- 无角色时是否有明确空状态提示
- 无角色时 `Start Task` 是否保持禁用或被正确限制
- 关闭后再次打开，界面状态是否正常

### 通过标准
- 弹窗交互自然，没有卡死、闪烁、空白块
- 空状态提示清楚
- 关闭路径一致，没有“点了没反应”的情况

### 失败表现
- 弹窗打不开或关闭异常
- 样式错位明显
- 无角色也能直接创建任务
- 再次打开后状态混乱

---

## 2. 验证 Add Role 弹窗的 archetype-first 体验

### 操作步骤
1. 在 Task Builder 中点击 `+ Add Role`。
2. 打开 Add Role 弹窗后，先不要提交。
3. 查看表单结构。
4. 依次切换 2~3 个不同 archetype。

### 需要观察的内容
- 表单是否以 archetype 选择为核心，而不是旧的固定 identity 选择
- 是否能明显看到以下信息区域：
  - Role Name
  - Archetype 选择
  - Provider / Model 选择
  - System Prompt Override
  - Handoff 开关
- 切换 archetype 后，预览信息是否随之变化
- 说明信息是否足够帮助你理解该角色适合做什么

### 通过标准
- archetype 是主要选择项
- 切换 archetype 时，说明内容明显变化
- 表单字段完整，没有缺失或歧义

### 失败表现
- 看起来仍然像旧版 identity 表单
- archetype 选择和详情区域脱节
- 切换不同 archetype 时展示内容几乎不变
- 字段命名让人难以理解

---

## 3. 验证 archetype 详情内容是否“可用”

### 操作步骤
1. 选择一个 archetype，认真阅读其详情。
2. 再切换到另一个 archetype，对比差异。
3. 判断这些信息是否足够支撑你做角色选择。

### 需要观察的内容
重点观察这些信息是否清楚、有区分度：
- summary / 概述
- responsibilities / 负责内容
- boundaries / 边界
- deliverables / 交付物
- recommended next archetypes / 推荐下游角色
- handoff guidance / 交接说明

### 通过标准
- 每个 archetype 的定位清楚，不混淆
- 你能根据说明判断“这个角色适不适合当前任务”
- 不同 archetype 之间有真实差异，而不是模板化重复文案

### 失败表现
- 说明过于空泛，看完仍不知道该怎么选
- 不同 archetype 文案高度重复
- 推荐下游角色没有参考价值
- handoff guidance 读起来不自然或不实用

---

## 4. 验证单角色任务创建后的信息展示

### 操作步骤
1. 创建一个仅包含 1 个角色的任务。
2. 角色配置建议：
   - Role Name：自定义一个容易识别的名字
   - Archetype：任选一个
   - Provider / Model：任选一个可用组合
   - 不填写 override
   - 不开启 handoff
3. 创建完成后，查看任务中的角色展示区域。

### 需要观察的内容
- 角色名是否与输入一致
- archetype 信息是否展示正确
- provider / model 展示是否正确
- 角色卡片信息是否易读
- 页面中是否还能明显辨认这个角色的身份和用途

### 通过标准
- 你创建时填的关键信息都能在界面中对上
- 没有“显示成别的角色”或“字段串位”的现象

### 失败表现
- 创建后角色信息丢失
- archetype 名称不对
- provider / model 显示混乱
- 卡片信息不完整或难以辨认

---

## 5. 验证多角色顺序与展示

### 操作步骤
1. 新建一个包含 3 个角色的任务。
2. 建议你按固定顺序添加，例如：
   - 角色 1：规划类角色
   - 角色 2：开发类角色
   - 角色 3：评审类角色
3. 创建任务后，查看任务中的角色列表。

### 需要观察的内容
- 任务创建前看到的顺序，与创建后是否一致
- `Role 1 / Role 2 / Role 3` 的编号是否符合直觉
- 列表展示是否清楚体现先后关系
- 顺序是否会出现错位、跳号、重复

### 通过标准
- 添加顺序 = 展示顺序
- 角色编号清晰稳定
- 你能一眼看出协作链路

### 失败表现
- 创建前后顺序不一致
- 编号错误或显示不稳定
- 视觉上看不出主次顺序

---

## 6. 验证 `system_prompt_override` 的可感知效果

### 操作步骤
1. 新建一个任务，添加 1 个角色。
2. 在 `System Prompt Override` 中输入一段非常容易观察的约束，例如：
   - `回答时必须先输出“执行计划：”四个字，然后给出三条编号步骤。`
3. 创建任务。
4. 进入该角色会话，发送一条简单消息，例如：
   - `请分析当前项目首页的改造思路。`
5. 观察回复格式。

### 需要观察的内容
- 回复是否明显受到 override 影响
- 输出格式是否体现你写入的约束
- 这种影响是稳定存在，还是偶发失效
- override 是否和 archetype 原本角色定位发生明显冲突

### 通过标准
- 回复中能明显看出 override 的存在
- override 是“增强角色行为”，不是让角色失真
- 至少在首轮回复里效果清楚可见

### 失败表现
- 完全看不出 override 生效
- 角色回复与 override 明显冲突
- override 一加入，角色行为反而变乱

---

## 7. 验证 `handoff_enabled` 的人工可见性

### 操作步骤
1. 创建一个 2~3 角色任务。
2. 其中至少一个角色开启 `handoff_enabled`，另一个保持关闭。
3. 创建任务后查看角色卡片。
4. 进入开启 handoff 的角色会话，发起一轮交流。

### 需要观察的内容
- 开启 handoff 的角色是否有明显标识
- 未开启的角色是否不会误显示该标识
- 这个标识是否足够明显、易理解
- 从产品感觉上看，用户是否能理解“这个角色允许给出交接建议”

### 通过标准
- 开关状态在界面上可区分
- 标识不误导、不含糊

### 失败表现
- 开与不开看不出区别
- 所有角色都显示一样
- 文案让人无法理解 handoff 的含义

---

## 8. 验证多角色切换体验

### 操作步骤
1. 进入一个多角色任务。
2. 依次切换不同角色。
3. 每个角色分别发一条非常短的问题，例如：
   - `你在这个任务中的职责是什么？`
4. 在角色之间来回切换数次。

### 需要观察的内容
- 当前激活角色是否清晰
- 切换后标题、角色卡片、高亮状态是否一致
- 是否出现“点的是 A，看到的是 B”
- 来回切换时会不会让人迷失上下文

### 通过标准
- 当前角色状态非常明确
- 每个角色的会话边界感清楚
- 切换体验稳定，不混乱

### 失败表现
- 视觉高亮和实际会话不一致
- 切换后还是上一个角色内容
- 使用几次后分不清当前是谁

---

## 9. 验证 archetype 对角色回答风格的影响

### 操作步骤
1. 创建一个多角色任务，选择差异明显的 archetype。
2. 给不同角色发送同一个问题，例如：
   - `请给出当前功能的下一步建议。`
3. 对比不同角色的回答。

### 需要观察的内容
- 不同角色的关注点是否不同
- 规划类角色是否更偏方案、范围、优先级
- 开发类角色是否更偏实现、技术细节
- 评审类角色是否更偏风险、边界、验证

### 通过标准
- 同一问题下，不同角色有明显风格差异
- 这种差异符合 archetype 预期

### 失败表现
- 不同角色回答几乎一样
- 角色人格区分度很弱
- archetype 选了但实际对输出影响不明显

---

## 10. 验证手动 Forward 流程没有被破坏

### 操作步骤
1. 在一个多角色任务中，让角色 A 先产生一条回复。
2. 使用现有的手动 forward / 转发入口。
3. 将角色 A 的结果转发给角色 B。
4. 查看角色 B 是否正确收到上下文，并继续回答。

### 需要观察的内容
- Forward 入口是否还在、是否易找
- 弹窗或选择流程是否顺畅
- 被转发目标是否明确
- 角色 B 收到的内容是否足够连贯
- 转发后整个链路是否自然，没有强烈割裂感

### 通过标准
- 手动 forward 全流程可用
- 转发后的上下文对目标角色是可理解的
- 没有因为 archetype 机制引入新的交互阻碍

### 失败表现
- Forward 入口消失或难用
- 转发目标选择混乱
- 转发后目标角色理解不了上下文
- 流程上出现明显卡顿或异常

## 五、建议你记录问题时使用的格式

如果你在验收中发现问题，建议按下面格式记录给我：

### 问题标题
一句话描述问题，例如：
- `多角色创建后顺序错乱`
- `override 已填写但回复没有体现`

### 复现步骤
1. 打开哪个页面
2. 做了什么操作
3. 输入了什么内容
4. 最终看到了什么异常

### 实际结果
描述你真实看到的现象。

### 预期结果
描述你原本认为应该发生什么。

### 附件
- 截图
- 录屏
- 报错原文

## 六、最小人工验收路径（时间不够时优先执行）

如果你时间有限，建议至少完成下面 6 项：

1. 打开 Task Builder，确认交互正常
2. 打开 Add Role，确认 archetype 选择和详情预览正常
3. 创建一个单角色任务，确认角色信息展示正确
4. 创建一个三角色任务，确认顺序稳定
5. 用一个带 override 的角色发消息，确认输出格式受影响
6. 做一次手动 forward，确认流程没坏

如果这 6 项都通过，本轮最关键的人工体验链路就基本通过了。
