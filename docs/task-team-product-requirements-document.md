**产品经理：** 管理员  
**优先级：** P1（中高优先级）  
**关联文档：** `docs/task-team-requirements.md`、`docs/task-team-development-plan.md`

## 1. 项目背景

### 1.1 背景说明

当前 Task AI 多角色团队协作能力已经支持：

- 为 task 配置多个角色（Role）
- 为每个角色绑定对应的大模型
- 基于角色 session 完成多轮协作与 handoff 交接
- 通过 task template 复用已有的团队配置

但现阶段 task template 的编辑能力主要集中在**角色绑定的大模型**层面，用户只能调整“某个角色使用哪个模型”，**无法调整 task 模板本身的成员组成**，例如：

- 新增一个新的团队成员
- 删除一个不需要的成员
- 修改成员顺序
- 调整不同成员在模板中的职责分工

这导致 task template 的可定制能力停留在“替换执行模型”层面，而没有真正开放“团队编排”能力。

### 1.2 当前问题

当前能力存在以下几个核心问题：

1. **模型配置与团队编排被混在一起**  
   用户可以改模型，但不能改成员，容易让用户误以为“模板已可编辑”，但实际上只开放了部分能力。

2. **自定义价值不足**  
   如果用户只能替换模型，不能控制团队成员组成，那么 task template 的复用价值和个性化空间有限。

3. **与 Task AI 的核心设计不完全一致**  
   在现有架构中，Task 的本质是“目标 + 团队配置”，而团队配置不仅包含模型绑定，也包含成员组成与角色关系。因此只允许改模型、不允许改成员，能力边界并不完整。

---

## 2. 产品目标

### 2.1 目标

本次需求旨在补齐 task template 的编辑边界，使用户能够在合理的权限范围内编辑团队成员组成。

具体目标如下：

1. 明确区分“模型配置”和“成员组成”是两层独立能力
2. 保持系统内置模板的稳定性和可预测性
3. 允许用户在自定义模板中编辑 task 的成员组成
4. 让用户可以基于系统模板复制出自己的模板，并在副本上进行团队编排调整

### 2.2 非目标

本次需求**不包含**以下内容：

1. 不改变 Task Session、Handoff、角色提示词拼接等底层机制
2. 不引入自动推荐成员、自动生成团队结构等智能编排能力
3. 不修改系统内置模板本身的只读属性
4. 不重新设计 archetype 来源、同步流程或 role-definition 文件结构

---

## 3. 核心产品规则

### 3.1 规则总述

本次需求建议将“是否允许编辑成员组成”的权限边界，绑定到**模板类型**，而不是绑定到**模型类型**。

### 3.2 模板类型与编辑权限

| 模板类型 | 是否允许编辑角色绑定模型 | 是否允许编辑成员组成 | 说明 |
|------|------|------|------|
| 系统内置模板 | 允许有限调整或按现有规则处理 | 不允许 | 保持系统预设团队结构稳定 |
| 用户自定义模板 | 允许 | 允许 | 用户拥有完整模板编排权限 |
| 基于系统模板复制出的用户模板 | 允许 | 允许 | 推荐作为主要扩展方式 |

### 3.3 成员组成的定义范围

“编辑成员组成”至少包括以下操作：

1. 新增成员
2. 删除成员
3. 调整成员顺序
4. 修改成员名称或角色定位
5. 为成员重新绑定模型
6. 调整成员是否参与当前模板协作

### 3.4 关键边界

需要明确以下边界：

- **模型是否为系统内置**，不决定模板是否可编辑成员
- **模板是否属于用户可管理对象**，才决定是否可编辑成员
- 系统模板本体保持只读
- 用户对系统模板的扩展，应通过“复制为自定义模板”完成，而不是直接改写系统模板

## 4. 概念澄清

### 4.1 模型配置

模型配置是指：某个角色最终绑定并调用哪个大模型执行任务。

示例：
- 产品经理角色绑定模型 A
- 前端开发角色绑定模型 B
- QA 角色绑定模型 C

### 4.2 成员组成

成员组成是指：一个 task template 中有哪些成员、每个成员扮演什么角色、它们如何构成协作团队。

示例：
- 团队是否包含 PM、前端、后端、QA
- 是否增加“架构师”角色
- 是否移除“设计师”角色
- 角色在团队中的顺序与组织方式

### 4.3 为什么两者必须分离

如果不区分这两层能力，会产生以下问题：

- 用户以为“我已经能编辑模板”，但实际上只能换模型
- 产品规则难以解释：为什么能新增模型，却不能让模型进入团队
- 后续扩展模板市场、模板复制、模板分享能力时，权限边界会混乱

因此，本产品需求明确要求：

> 模型配置是角色级能力，成员组成是模板级能力，两者必须分别管理。

## 5. 用户场景

### 5.1 场景一：复制系统模板后裁剪团队

用户选择一个系统内置的“全栈开发团队”模板，复制为自己的模板后：

- 删除暂时不需要的 QA 成员
- 将“前端开发”替换为自己常用模型绑定的角色
- 保留 handoff 协作机制不变

### 5.2 场景二：基于自定义模型扩展团队

用户已经新增了自己定义的模型，希望在某个 task template 中：

- 增加一个“代码审查员”成员
- 让该成员绑定用户自定义模型
- 参与团队协作流程

此时用户需要的不只是“选择模型”，而是“把这个成员正式加入模板结构”。

### 5.3 场景三：针对不同任务维护不同团队模板

用户希望维护多套模板：

- 一套用于快速原型：PM + 前端
- 一套用于标准开发：PM + 前端 + 后端 + QA
- 一套用于重构审查：架构师 + 后端 + Reviewer

这些需求都依赖对成员组成的编辑能力，而不仅仅是模型替换能力。

## 6. 功能需求

### 6.1 模板编辑能力

对于用户自定义模板，系统应支持以下操作：

1. 查看当前模板的全部成员列表
2. 新增成员到模板中
3. 删除模板中的成员
4. 调整成员顺序
5. 修改成员的名称、职责描述或 archetype 绑定
6. 修改成员绑定的模型
7. 保存模板修改结果

### 6.2 系统模板限制

对于系统内置模板，系统应满足以下要求：

1. 用户可以查看系统模板的成员组成
2. 系统模板本体不可直接编辑成员组成
3. 若用户希望调整成员组成，系统应提供“复制为自定义模板”入口
4. 用户在复制出的模板中获得完整编辑权限

### 6.3 创建与复制流程要求

| 场景 | 期望行为 |
|------|------|
| 用户新建自定义模板 | 默认允许编辑成员组成 |
| 用户复制系统模板 | 生成新的用户模板副本，允许编辑成员组成 |
| 用户打开系统模板详情 | 展示只读状态 |
| 用户尝试修改系统模板成员 | 给出不可直接编辑的提示，并引导复制 |

### 6.4 权限反馈要求

系统在 UI 和交互上应清晰表达以下状态：

- 当前模板是系统模板还是用户模板
- 当前模板的成员组成是否可编辑
- 不可编辑的原因是什么
- 如果不可编辑，下一步可执行动作是什么（例如复制模板）

---

## 7. 交互与文案要求

### 7.1 编辑入口要求

在模板编辑界面中，应分别体现：

- 成员组成管理区域
- 角色模型绑定区域

避免让用户误以为修改模型就是修改团队结构。

### 7.2 只读状态要求

对于系统模板，应显示明确的只读提示，例如：

- “该模板为系统内置模板，成员组成不可直接修改”
- “如需调整团队成员，请先复制为自定义模板”

### 7.3 复制引导要求

在系统模板界面中，应提供明确入口：

- 复制模板
- 基于当前模板创建自定义版本
- 复制后进入可编辑状态

---

## 8. 验收标准

### 8.1 功能验收

- [ ] 用户可以识别模板是系统内置还是用户自定义
- [ ] 用户自定义模板支持新增、删除、排序成员
- [ ] 用户自定义模板支持修改成员绑定模型
- [ ] 系统内置模板不可直接修改成员组成
- [ ] 系统提供从系统模板复制为自定义模板的能力或明确入口
- [ ] 复制后的模板可以编辑成员组成

### 8.2 认知验收

- [ ] 用户能够理解“模型配置”和“成员组成”是两类不同配置
- [ ] 用户不会误以为“能换模型”就等于“能编辑团队模板”
- [ ] 用户能清楚理解为什么系统模板不能直接改，但复制后可以改

### 8.3 一致性验收

- [ ] 该规则与现有 Task / Role / Task Template 概念保持一致
- [ ] 不破坏现有 Task Session 与 Handoff 工作流
- [ ] 不引入与现有系统模板资源管理方式冲突的产品规则

---

## 9. 与现有架构的关系说明

根据当前文档定义：

- Task 是“任务 + 团队配置”的工作单元
- Role 是团队中的一员
- 系统提示词中已经明确包含团队成员编号、角色名称和团队配置

因此，从产品定义上看，**成员组成本来就是 task template 的核心组成部分**，而不仅是附属信息。

本次需求不是新增一个完全陌生的能力，而是把现有 Task AI 的“团队配置”概念补齐到模板编辑层。

也就是说，本次调整属于：

- **产品能力边界补全**，而不是机制重构
- **模板编辑权限优化**，而不是协作引擎改造

---

## 10. 产品结论

本需求的正式产品结论如下：

> 系统内置 task 模板保持只读，确保预设团队结构稳定；用户创建的模板，以及基于系统模板复制得到的自定义模板，应允许用户编辑成员组成，包括新增、删除、排序成员及调整模型绑定。

该规则比“只有用户自定义模型才能修改 task 成员组成”更清晰，也更稳定，因为它将权限边界建立在**模板归属**之上，而不是建立在**模型来源**之上。

这将使 Task AI 的模板体系更符合用户认知，也更符合后续模板复制、模板市场与团队编排能力的扩展方向。
