# 测试手册 — Phase 3 运行时角色管理 + 回归验证

**测试日期**: 2026-05-09  
**测试范围**: Phase 3 新增的 4 个后端 handler（运行时角色增/删/改/排序）+ 回归验证  
**预计时间**: 15-20 分钟

---

## 说明

本轮新增了后端 4 个 Tauri 命令，对应前端已有的角色管理 UI：

| 命令 | 前端入口 |
|------|---------|
| `add_task_role` | 角色列表底部「+ 添加角色」按钮 |
| `update_task_role` | 角色卡片右上角「编辑」图标 |
| `delete_task_role` | 角色卡片右上角「删除」图标 |
| `reorder_task_roles` | 拖拽角色卡片调整顺序 |

---

## 准备工作

1. 启动开发服务器：`npm run dev`
2. 启动 Tauri 桌面端
3. 打开开发者控制台（查看日志和错误）
4. 先创建一个 Task（如果还没有），确保 Task 中有至少 2 个角色

---

## 测试项

### B1: 运行中任务 — 添加角色

**操作步骤**:
1. 打开一个已有的 Task，进入角色列表视图
2. 点击「+ 添加角色」
3. 填写角色名称（如 "Reviewer"）
4. 选择身份类型（identity）
5. 选择 Provider（如 openai）
6. 点击确认

**预期效果**:
- 提示"成功添加角色"
- 角色列表中出现新角色
- 新角色的 provider 显示正确
- 控制台无报错

**通过标准**: 新角色出现在列表中，provider 信息正确。

---

### B2: 运行中任务 — 编辑角色

**操作步骤**:
1. 在角色列表中，点击某个角色的「编辑」图标
2. 修改 Name
3. 修改 Provider（换一个不同的 provider）
4. 点击确认

**预期效果**:
- 提示更新成功
- 角色卡片即时刷新，显示新的 name 和 provider
- 控制台无报错

**通过标准**: 角色卡片显示修改后的值，无需手动刷新页面。

---

### B3: 运行中任务 — 删除角色

**操作步骤**:
1. 在角色列表中，点击某个角色的「删除」图标
2. 确认删除对话框弹出
3. 点击确认

**预期效果**:
- 提示"成功删除角色"
- 角色列表中该角色消失
- 控制台无报错

**通过标准**: 角色被移除，列表即时更新。

---

### B4: 运行中任务 — 拖拽排序角色

**操作步骤**:
1. 在角色列表中（至少 3 个角色）
2. 鼠标按住某个角色卡片拖动
3. 拖到新位置后松开

**预期效果**:
- 角色卡片的顺序发生变化
- 新顺序被保存（关闭 Task 再打开后顺序不变）
- 控制台无报错

**通过标准**: 拖拽后顺序保持，重开 Task 后顺序不变。

---

### B5: 重启后角色数据持久化

**操作步骤**:
1. 执行 B1 或 B2（添加或编辑一个角色）
2. 完全退出应用
3. 重新启动应用
4. 打开同一个 Task

**预期效果**:
- 之前添加/编辑的角色仍然存在
- Provider、name 等字段与保存时一致
- 排序（如执行过 B4）保持不变

**通过标准**: 数据在应用重启后完全保留。

---

### B6: 回归 — 模板编辑 + provider 保存

**操作步骤**:
1. 进入模板列表
2. 点击某个模板的「Edit」
3. 修改某个角色的 provider
4. 保存
5. 退出编辑后重新打开该模板

**预期效果**:
- 保存成功，无报错
- 重新打开后 provider 修改仍然保留
- 卡片状态正确刷新

**通过标准**: 之前完成的模板编辑功能仍然正常工作。

---

### B7: 回归 — Save Task as Template

**操作步骤**:
1. 打开一个有角色的 Task
2. 点击「保存为模板」
3. 输入模板名称和描述
4. 保存后回到模板列表

**预期效果**:
- 新模板出现在用户模板列表中
- 新模板可以打开详情
- 角色的 provider 信息正确保留

**通过标准**: 另存为模板功能正常工作。

## 常见问题

### Q1: 添加角色时报 "command not found"
- 确认后端已重新编译
- 检查控制台是否有函数未注册的错误

### Q2: 拖拽排序后顺序没保存
- 打开控制台查看 `reorder_task_roles` 是否调用成功
- 检查网络请求是否返回错误

### Q3: 重启后数据丢失
- 检查后端是否正确写入了 SQLite
- 查看 `~/.local/share/com.your-app/` 下数据库文件是否更新

### Q4: 回归测试失败
- 检查 `update_template` 和 `save_task_as_template` 是否仍正常注册
- 查看前端 API 调用是否有变动
