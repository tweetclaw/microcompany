# 多 Agent 协作编排系统 - 需求文档

## 1. 项目背景

当前系统是一个单 Session 的 AI 对话应用，用户通过配置不同的大模型进行对话。现需要升级为**多 Agent 协作编排系统**，实现多个 AI Agent 之间的协作，模拟真实软件开发团队的工作流程。

## 2. 核心需求

### 2.1 角色与团队配置

**团队角色定义**：
- **项目经理（PM）**：负责需求分析、任务分解、进度跟踪、结果汇总
- **开发人员（Developer）**：负责编写代码实现
- **评审员（Reviewer）**：负责代码评审、方案评审
- **测试人员（Tester）**：负责测试验证

**配置要求**：
- 每个角色可配置独立的大模型（Provider + Model）
- 每个角色可配置专属的 System Prompt（角色职责描述）
- 支持预设团队模板（如"标准开发团队"、"快速原型团队"等）
- 团队配置可保存、加载、复用

### 2.2 Session 层级架构

**主 Session（Master Session）**：
- 用户直接交互的对话窗口
- 由项目经理角色（Alice）担任
- 负责理解用户需求、编排子任务、汇总结果

**子 Session（Sub Session）**：
- 由主 Session 创建和管理
- 每个子 Session 对应一个特定角色
- 拥有独立的对话历史和上下文
- 执行完任务后向主 Session 汇报结果

### 2.3 工作流编排

**典型开发流程示例**：

```
用户 → Alice (PM)
  ↓
  Alice 分析需求 → 生成开发文档
  ↓
  用户确认方案
  ↓
  Alice 调度 Bob (Developer) → 编写开发计划
  ↓
  Bob 完成 → 汇报给 Alice
  ↓
  Alice 调度 Clark (Reviewer) → 评审开发计划
  ↓
  Clark 发现问题 → 反馈给 Alice
  ↓
  Alice 转发给 Bob → Bob 修改
  ↓
  Bob 修改完成 → 汇报给 Alice
  ↓
  Alice 再次调度 Clark → 评审通过
  ↓
  Alice 调度 Dark (Developer) → 开始开发
  ↓
  Dark 开发完成 → 汇报给 Alice
  ↓
  Alice 调度 Bob & Clark → 代码评审
  ↓
  评审通过 → Alice 汇报给用户
  ↓
  用户手动测试 → 反馈问题或确认完成
```

**编排规则**：
- 主 Session 可以创建、暂停、恢复、终止子 Session
- 子 Session 之间可以通过主 Session 进行消息传递
- 支持串行执行（A 完成后执行 B）和并行执行（A 和 B 同时执行）
- 支持条件分支（如果评审通过则继续，否则返回修改）
- 支持循环迭代（评审-修改-再评审）

### 2.4 消息路由与通信

**消息类型**：
- **用户消息**：用户发送给主 Session
- **任务分配消息**：主 Session 发送给子 Session
- **结果汇报消息**：子 Session 发送给主 Session
- **转发消息**：主 Session 在子 Session 之间转发

**路由机制**：
- 每个消息携带 `from`（发送者）和 `to`（接收者）标识
- 主 Session 维护消息队列和路由表
- 支持消息历史追踪（谁在什么时候说了什么）

### 2.5 上下文管理

**主 Session 上下文**：
- 用户需求描述
- 当前任务状态
- 所有子 Session 的汇报结果
- 工作流执行历史

**子 Session 上下文**：
- 角色定义（System Prompt）
- 分配的任务描述
- 任务执行过程中的对话历史
- 需要的上下文信息（由主 Session 提供）

**上下文传递**：
- 主 Session 创建子 Session 时，传递必要的上下文
- 子 Session 完成任务时，返回结果摘要
- 支持上下文压缩（避免子 Session 上下文过大）

### 2.6 UI/UX 需求

**团队配置界面**：
- 可视化配置团队成员（角色、模型、Prompt）
- 支持添加/删除/编辑角色
- 支持导入/导出团队配置

**对话界面增强**：
- 主对话区：显示用户与 Alice 的对话
- 子任务面板：显示当前活跃的子 Session 及其状态
- 工作流可视化：显示任务流转路径（类似流程图）
- 消息溯源：点击消息可查看完整的传递路径

**状态指示**：
- 每个子 Session 的状态（等待中、执行中、已完成、失败）
- 当前工作流进度（第几步/共几步）
- 实时显示哪个 Agent 正在工作

## 3. 非功能需求

### 3.1 性能要求
- 支持至少 10 个并发子 Session
- 消息路由延迟 < 100ms
- 主 Session 响应时间 < 2s

### 3.2 可靠性要求
- 子 Session 失败时，主 Session 可以重试或切换策略
- 支持工作流中断后恢复
- 所有对话历史持久化存储

### 3.3 可扩展性要求
- 支持自定义角色类型
- 支持自定义工作流模板
- 支持插件机制（未来可扩展更多 Agent 能力）

## 4. 约束条件

- 必须兼容现有的 Provider 配置系统
- 必须兼容现有的 Session 存储机制
- 不能破坏现有的单 Session 使用模式（向后兼容）
- 遵守 CLAUDE.md 中的文件操作协议和子模块保护协议

## 5. 验收标准

1. 用户可以配置一个包含 4 个角色的开发团队
2. 用户可以通过主 Session 发起一个开发任务
3. 系统自动按照工作流编排，依次调度各个角色完成任务
4. 用户可以在 UI 中看到完整的任务流转过程
5. 所有对话历史可以追溯和导出
6. 系统在子 Session 失败时能够优雅降级或重试

## 6. 里程碑

- **Phase 1**：团队配置系统 + 基础 Session 管理
- **Phase 2**：消息路由 + 简单工作流编排（串行执行）
- **Phase 3**：复杂工作流编排（并行、条件、循环）
- **Phase 4**：UI 可视化 + 工作流模板系统
