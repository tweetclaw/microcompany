# Task 角色调度与结果转发 - UI/UX 最终设计文档

## 1. 最终推荐方案摘要

本方案的核心目标，是在**不修改现有聊天系统**的前提下，增加一个轻量的 Task 工作台，用来组织多个角色 session，并支持用户手动调度结果流转。

### 最终推荐 UI 方案
- 在现有工具栏中新增 `New Task` 入口
- 创建 Task 后，先进入角色配置页
- 用户逐个创建角色，并为每个角色指定名称、身份、Provider、Model
- 每个角色创建完成后，与一个对应 session 建立关联
- 完成角色配置后，进入 Task 调度页
- 在调度页中：
  - 左侧显示当前 Task 的角色列表
  - 顶部显示当前正在对话的角色
  - 中间直接复用现有聊天界面
  - 顶部提供 `Forward Latest Reply` 按钮
- 用户可以把当前角色最新一条 AI 回复，连同补充说明一起，转发给同一 Task 内的其他角色

### 这个方案为什么成立
因为它最大限度复用了现有系统：
- 不重做 chat UI
- 不推翻 session 机制
- 不引入自动多 Agent 编排
- 只增加一个轻量但清晰的“任务组织 + 手动调度”层

---

## 2. 设计目标与边界

### 2.1 设计目标
本次只新增一个轻量的 Task 工作台，用来把多个角色 session 组织起来，并支持手动转发结果。

用户可以：
- 创建一个 Task
- 在 Task 中添加多个角色
- 为每个角色指定名称、身份和大模型
- 为每个角色生成对应的 session
- 在 Task 中切换角色并继续使用现有聊天能力
- 将当前角色最新一条 AI 回复转发给同一 Task 内其他角色
- 在转发时附加说明，或直接转发

### 2.2 本次明确不做
本次设计不涉及以下内容：
- 不修改现有 AI 对话框逻辑
- 不重做现有聊天界面
- 不改变现有 session 的基本运行方式
- 不实现自动工作流编排
- 不实现角色之间自动通信
- 不实现消息流可视化
- 不支持任意历史消息转发
- 不支持批量转发给多个角色

---

## 3. 开发时必须坚持的 5 条原则

1. **Task 是外壳，不是新的聊天系统**
   - Task 只负责组织、切换、转发
   - 聊天区继续复用现有能力

2. **角色切换要像现有 session 切换一样自然**
   - 用户不应该学习一套新的切换逻辑

3. **转发只围绕“当前角色最新回复”展开**
   - 第一版不引入历史消息选择，避免复杂化

4. **失败状态必须就地处理**
   - 创建失败和转发失败都不应让用户丢失输入

5. **界面始终要让用户知道当前在和谁工作**
   - 当前角色信息必须持续可见

## 4. 用户任务流程

### 流程 A：创建 Task
1. 用户点击 `New Task`
2. 进入 Task 创建页
3. 初始界面为空，提示先添加角色

### 流程 B：添加角色
1. 用户点击 `Add Role`
2. 填写角色信息：
   - 角色名称
   - 角色身份
   - Provider
   - Model
   - 可选系统提示词
3. 点击 `Create Role`
4. 系统为该角色创建对应 session
5. 角色出现在当前 Task 的角色列表中
6. 用户继续添加其他角色

### 流程 C：进入 Task 调度
1. 用户完成角色配置
2. 点击 `Start Task`
3. 进入 Task 调度页

### 流程 D：与角色对话
1. 用户在 Task 中选择一个角色
2. 主聊天区切换到该角色对应的 session
3. 用户继续像现在一样与该角色对话
4. 页面顶部始终显示当前对话角色

### 流程 E：转发当前角色最新结果
1. 当前角色产生新的 AI 回复
2. 用户点击 `Forward Latest Reply`
3. 选择当前 Task 内其他角色作为目标
4. 可填写补充说明，也可留空
5. 点击确认
6. 系统将“补充说明（可选） + 当前角色最新一条 AI 回复”发送到目标角色 session
7. 用户按需切换到目标角色继续工作

## 5. 与现有系统的关系

### 5.1 Task 不是新的聊天系统
Task 只是现有聊天系统外的一层组织壳。

它负责：
- 组织角色
- 在角色之间切换
- 提供转发入口

它不负责：
- 重做聊天区
- 重做消息渲染
- 重做发送逻辑

### 5.2 聊天区直接复用现有界面
Task 调度页中的主聊天区域，应该直接复用当前聊天界面。

用户体验上：
- 选中某个角色
- 就像当前从列表中切换某个 session
- 中间聊天区显示该角色对应的现有 session

因此，Task 页本质上是：
- 左侧：角色导航
- 顶部：当前角色信息 + 转发入口
- 中间：现有聊天界面

### 5.3 角色切换体验应接近现有 session 切换
- 角色列表像任务内的 session 导航
- 选中角色后立即切换会话内容
- 不引入新的复杂状态机

## 6. 入口与信息架构建议

### 6.1 入口位置建议
基于当前产品结构，`New Task` 应作为一级入口放在现有“新建”动作附近。

推荐优先位置：
- 顶部工具栏中的 `New Task`

推荐原因：
- 当前 `Toolbar` 已经承载创建行为
- 用户容易理解：`New Session` 与 `New Task` 是并列入口
- 不会打断当前 session 导航结构

### 6.2 与现有导航的关系
当前产品已经有稳定的左侧导航心智模型，因此 Task 内部的角色列表建议延续这种模式。

换句话说：
- 创建入口可以在工具栏
- 进入 Task 后，角色浏览与切换逻辑应贴近现有侧边列表体验

## 7. 页面与组件设计

## 7.1 页面一：Task 创建 / 角色配置页

### 页面目标
让用户先完成当前任务所需角色的配置。

### 页面示意

```text
┌─────────────────────────────────────────────────────────────┐
│  New Task                                                   │
├─────────────────────────────────────────────────────────────┤
│  Task Name:                                                 │
│  [登录功能方案协作任务                                  ]   │
│                                                             │
│  Roles                                                      │
│                                                             │
│  No roles yet. Create the first role for this task.         │
│                                                             │
│  [+ Add Role]                                               │
│                                                             │
│                                    [Cancel] [Start Task]    │
└─────────────────────────────────────────────────────────────┘
```

### 页面规则
- 至少有一个角色后，`Start Task` 才可点击
- 角色列表需要明确表达：该角色不是草稿，而是已与 session 关联
- 配置页只关注“把角色建好”，不在这里引入转发逻辑

### 页面状态

#### 空状态
- 页面初始只强调“先添加第一个角色”
- `Start Task` 默认禁用

#### 可操作状态
- 至少存在一个角色时，`Start Task` 可点击
- 角色卡片正常展示已配置内容

#### 提交中状态
- 用户点击 `Create Role` 后，按钮进入 loading 状态
- 防止重复点击

#### 失败状态
- 错误提示应在弹层内就地显示
- 原因说明尽量直接，如：模型未选择、角色名为空、session 创建失败

快速创建一个角色，并与一个 session 关联。

### 表单规则
- 角色名称必填
- 角色身份必填
- Provider 必填

### 状态规则

- 创建中时按钮 loading

- 创建成功后关闭弹层，并在角色列表中显示 `Session ready`

## 7.3 页面二：Task 调度页

### 页面目标
让用户在一个 Task 内：
- 浏览所有角色
- 切换到某角色对应的 session
- 把当前角色最新结果转给其他角色

### 页面示意

```text
┌────────────────────────────────────────────────────────────────────────────┐
│  Task: 登录功能方案协作任务                                               │
├──────────────────────┬─────────────────────────────────────────────────────┤
│ Roles in Task        │ Current Role: Alice (Product Manager)             │
│                      ├─────────────────────────────────────────────────────┤
│  ● Alice             │ [Forward Latest Reply]                             │
│    Product Manager   │                                                     │
│                      │ ┌───────────────────────────────────────────────┐   │
│  ○ Bob               │ │                                               │   │
│    Reviewer          │ │      Existing Chat UI / Existing Session      │   │
│                      │ │            (reused as-is)                     │   │
│  ○ Clark             │ │                                               │   │
│    Developer         │ │                                               │   │
│                      │ └───────────────────────────────────────────────┘   │
│                      │                                                     │
└──────────────────────┴─────────────────────────────────────────────────────┘
```

### 页面原则
1. 聊天区不重做，只复用现有聊天界面
2. 角色列表用于快速切换
3. 顶部必须明确显示当前角色
4. 转发入口围绕“当前角色最新结果”设计
5. 用户始终是调度者，不做自动协作

### 页面状态

#### 首次进入状态
- 默认选中第一个可用角色，或选中用户最后创建的角色
- 中间聊天区立即切换到对应 session
- 当前角色信息同步显示

#### 正常工作状态
- 用户可以自由切换角色
- 切换后页面头部与聊天区同步变化
- 转发按钮状态根据当前角色是否存在最新 assistant 回复实时变化

#### 空聊天状态
- 如果当前角色 session 尚无消息，聊天区保持现有空态样式
- `Forward Latest Reply` 按钮禁用

#### 切换中状态
- 如果角色切换需要加载 session 内容，可显示轻量 loading
- 不应阻塞整个 Task 页面

## 7.4 当前角色信息区

当前角色信息必须始终可见，避免用户在多个角色之间迷失。

推荐显示方式：

```text
Current Role: Alice (Product Manager)
```

也可以扩展为两行：

```text
Current Role: Alice
Identity: Product Manager
```

设计要求：
- 视觉层级高于聊天内容
- 角色切换时即时更新
- 与转发按钮放在同一区域，形成清晰的任务操作头部

---

## 7.5 转发入口

### 推荐按钮文案
- `Forward Latest Reply`

### 为什么不放在每条消息上
因为第一版只支持转发“当前角色最新一条 AI 回复”，不需要引入消息级选择器。

### 按钮状态
- 当前角色还没有 AI 回复：禁用
- 当前角色已有最新 AI 回复：可点击
- 当前角色正在生成回复：第一版建议禁用，避免转发未完成结果

- 补充说明不是必填项
- 用户可以直接点击 `Forward`
- 目标角色不能是自己
- 目标列表只显示当前 Task 内其他角色

### 状态规则
- 默认带出当前角色最新回复预览
- 如果 Task 中只有一个角色，转发入口应禁用，或明确提示无可转发目标
- 点击 `Forward` 后按钮 loading，防止重复提交
- 转发失败时保留目标与补充说明，不关闭弹层
- 转发成功后关闭弹层，并给出轻量反馈，如 `Forwarded to Bob`
- 默认不自动切换目标角色

## 7.7 角色列表体验

角色列表在 Task 中应承担“任务内 session 导航”的作用。

### 基本样式
- 当前角色高亮
- 其他角色普通态
- 每项展示：角色名 + 身份

例如：

```text
● Alice
  Product Manager

○ Bob
  Reviewer

○ Clark
  Developer
```

### 第一版不强制展示的信息
- 最近活跃时间
- 未读计数
- 是否已收到转发内容
- 复杂状态标记

### 异常保护
如果出现异常数据状态，页面应：
- 显示空状态提示
- 引导用户返回角色配置或重新创建角色
- 不展示无意义的聊天区和转发操作

---

## 7.8 响应式与窄屏行为

### 桌面宽屏
- 左侧显示角色列表
- 右侧显示当前角色头部 + 现有聊天区
- 转发按钮固定在头部区域

### 中等宽度
- 角色列表可收窄
- 当前角色信息与转发按钮保持在聊天区上方

### 窄屏 / 小窗口
- 角色列表可折叠为抽屉
- 当前角色头部固定在聊天区顶部
- 转发弹层保持全屏或大尺寸抽屉表现

核心原则：
- 不牺牲“当前正在和谁对话”的可感知性
- 不隐藏转发能力到过深层级
- 不为了适配小屏而改变核心交互模型

## 8. 组件清单与职责

### 8.1 Task 入口组件
**名称建议**：`New Task`

**职责**：
- 提供进入 Task 创建流程的一级入口

### 8.2 Task 创建页容器
**职责**：
- 展示 Task 名称输入区域
- 承载角色列表
- 承载 `Add Role` 与 `Start Task` 操作

### 8.3 角色卡片组件
**职责**：
- 展示单个角色基础信息
- 展示身份、Provider、Model
- 展示 session 是否已就绪

**职责**：
- 收集角色配置

- 在失败时保留输入并允许重试

### 8.5 Task 调度页容器
**职责**：
- 承载整个 Task 工作台
- 管理当前选中角色
- 驱动聊天区切换到对应 session
- 承载当前角色头部与转发入口

### 8.6 角色列表组件
**职责**：
- 展示当前 Task 内所有角色
- 支持切换当前工作角色

### 8.7 当前角色头部组件
**职责**：
- 持续显示当前对话角色
- 展示角色身份
- 承载 `Forward Latest Reply` 操作

### 8.8 Forward 弹层组件
**职责**：
- 展示当前角色最新 AI 回复预览
- 选择目标角色
- 输入补充说明
- 确认转发并反馈结果

### 8.9 聊天承载区域
**职责**：
- 直接承载现有聊天界面
- 随角色切换显示不同 session

## 9. 文案与命名规范建议

### 9.1 按钮文案建议
推荐统一使用：
- `New Task`
- `Add Role`
- `Create Role`
- `Start Task`
- `Forward Latest Reply`
- `Forward`
- `Cancel`

原则：
- 动词开头
- 短而明确
- 同一动作只保留一种叫法

### 9.2 状态文案建议
推荐使用：
- `No roles yet. Create the first role for this task.`
- `Session ready`
- `No other roles available in this task`
- `Forwarded to Bob`

### 9.3 错误文案建议
推荐风格：
- `Role name is required`
- `Please select a model`
- `Failed to create role session`
- `Failed to forward latest reply`

原则：
- 不要只写“操作失败”
- 尽量告诉用户哪里出了问题
- 能贴近字段就贴近字段

### 9.4 成功反馈文案建议
推荐使用：
- `Role created`
- `Session ready`
- `Forwarded to Bob`

原则：
- 反馈轻量
- 不阻断当前操作
- 不使用过长提示文案

### 9.5 命名一致性建议
建议统一使用以下概念：
- `Task`：任务容器
- `Role`：任务中的角色
- `Session`：角色关联的实际会话
- `Forward`：转发当前角色最新 AI 回复

避免混用：
- Task / Workflow / Project
- Role / Agent / Member
- Session / Chat / Conversation

## 10. 实现约束与设计注意事项

### 10.1 现有聊天区应直接复用
当前产品已经有完整的聊天工作区，因此 Task 调度页不应重做聊天框，而应直接承载现有聊天界面。

### 10.2 现有 session 生命周期存在约束
当前产品的新会话创建方式，更接近“先选择模型，再在首条消息发送时初始化真实 session”。

而本设计中的产品目标是：
- 创建角色后，用户感知到该角色已经拥有对应 session

因此，文档在产品层面保持“角色创建后 session ready”的体验目标，但实现时需要适配现有 session 生命周期。

### 10.3 入口与导航应贴合现有结构
当前已有清晰的信息架构：
- 顶部工具栏承担创建动作
- 左侧列表承担切换导航

因此 Task 相关设计不应另起一套全新的导航系统，而应顺着现有结构扩展。

### 10.4 保持 Task 为轻量外壳
Task 的价值在于“组织和调度”，而不是替代底层对话系统。

后续如果继续扩展，也应优先保持：
- 复用现有 session 能力
- 少引入跨层复杂性
- 让用户始终知道自己在和哪个角色工作

## 12. 验收标准

- [ ] 用户可以从现有界面进入 `New Task`
- [ ] 用户可以创建一个 Task
- [ ] 用户可以在 Task 中添加多个角色
- [ ] 每个角色都可以与一个对应 session 建立关联
- [ ] 用户可以点击 `Start Task` 进入 Task 调度页
- [ ] 用户可以在 Task 调度页中看到所有角色
- [ ] 用户可以切换角色，并在中间区域继续使用现有聊天界面
- [ ] 界面可以明确显示当前正在对话的角色
- [ ] 用户可以将当前角色最新一条 AI 回复转发给同一 Task 内其他角色
- [ ] 转发时可以填写说明，也可以留空
- [ ] 当前角色没有最新 AI 回复时，转发按钮处于禁用状态
- [ ] 当前角色正在生成回复时，转发按钮可被正确禁用
- [ ] 转发成功后，用户获得明确反馈，且界面默认不自动跳转
- [ ] Add Role 与 Forward 弹层在失败时可以保留用户输入并允许重试
- [ ] 窄屏下仍能清楚感知当前角色和转发入口

## 13. 结论

这个方案把复杂的多 Agent 协作问题，收敛成一个清晰且可落地的 MVP：

- 用 Task 组织角色
- 用角色关联现有 session
- 用现有聊天区承载对话
- 用“转发最新 AI 回复”完成跨角色协作

它不试图推翻现有聊天系统，只是在现有产品结构上增加一个简单、明确、可控的任务调度层。
