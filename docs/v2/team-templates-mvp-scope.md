# Team Templates MVP 范围说明

**文档版本**: v1.0  
**创建日期**: 2026-05-03  
**项目**: MicroCompany  
**文档性质**: MVP 范围与实施说明

---

## 1. 背景

当前用户在创建多角色 task 时，存在明显重复劳动：

- 每次都要手动添加角色
- 每次都要重新选择 archetype
- 每次都要重新配置 provider/model
- 每次都要重复搭建相似的团队结构

这导致两类核心问题：

1. **创建效率低**：高频用户在每次新建 task 时都要重复完成相同配置
2. **有效团队无法沉淀**：一次已经验证有效的团队结构，无法转化为后续可复用资产

Team Templates 的目标不是改变现有 task 运行时模型，而是：

> **让用户把一组可复用的 task 团队结构保存为模板，并在创建新 task 时直接从模板开始。**

---

## 2. MVP 目标

本期 MVP 只解决两个最高价值场景：

1. **从模板快速创建 task**
2. **将当前 task 的团队结构保存为模板，供后续复用**

MVP 的成功标准是：

- 用户不必每次重新逐个添加角色
- 用户可以复用系统预置团队结构
- 用户可以沉淀自己的常用团队结构
- 所有运行时 role 在创建前都完成 provider/model 补齐

---

## 3. MVP 范围

### 3.1 本期必须交付

#### A. 系统模板只读能力

支持内置系统模板的读取与展示。

**包含内容**：
- 系统模板数据结构
- 系统模板列表查询
- 模板详情预览
- 模板角色信息展示：
  - 角色顺序
  - 角色名称
  - 推荐 archetype
  - 可选 provider/model

**说明**：
- 系统模板默认可以不配置 provider/model
- 系统模板是创建 task 的起点，不是可直接运行的 task

---

#### B. 从模板生成 Task Draft

用户在新建 task 时，可以选择“从模板创建”，系统会先将模板复制为可编辑的 task draft。

**draft 可编辑字段**：
- task 名称
- role 名称
- role archetype
- role provider/model

**固定原则**：
- 模板不直接创建 task
- 模板必须先进入 draft，再由用户确认创建

---

#### C. 创建前确认页（必经）

所有从模板创建 task 的流程，都必须进入创建前确认界面。

**确认页职责**：
- 展示即将创建的 task 配置
- 允许用户统一检查并修改每个 role 的配置
- 检查 provider/model 是否完整
- 对缺失项进行阻断

**校验规则**：
- 任一 role 缺少 provider 或 model，则禁止创建 task
- 所有 role 配置完整后，才允许调用现有 `create_task` 流程

**这是本期 MVP 的硬门槛。**

---

#### D. 保存当前 Task 为用户模板

用户可以将当前 task 的团队结构保存为用户模板，用于后续复用。

**保存入口**：
- 在已配置好的 task 中提供“保存为模板”操作

**保存内容**：
- 模板名称（必填）
- 模板描述（可选）
- 角色顺序
- 角色名称
- archetype
- provider/model（可选，若 task 中有配置则一并保存）

**固定原则**：
- 只保存团队结构，不保存对话历史
- 保存后与原 task 脱钩
- 原 task 修改不自动回写模板
- 模板修改也不影响既有 task

---

#### E. 用户模板可用于再次创建 Task

用户保存后的模板，必须能在新建 task 时被重新选择并复用。

**包含内容**：
- 用户模板列表展示
- 与系统模板并列出现在“从模板创建”入口中
- 选择用户模板后，仍进入同一套 draft + 创建前确认流程

---

## 4. 本期明确不做

为了避免范围失控，以下内容不纳入 MVP：

### 4.1 用户模板编辑
- 不支持直接编辑已保存模板
- 若需调整，可从 task 重新保存为新模板

### 4.2 用户模板完整管理能力
- 不做搜索
- 不做分类
- 不做标签
- 不做收藏
- 不做模板市场
- 不做版本历史

### 4.3 运行时动态团队管理
以下能力属于 task 运行时团队演化，不纳入本次 Team Templates MVP：
- 动态添加角色
- 动态删除角色
- 动态更新角色顺序/名称/handoff 配置

### 4.4 模板自动工作流
- 不做模板级 handoff 编排
- 不做模板自动协作路径
- 不做自动执行流程

### 4.5 复杂交互增强
- 不要求首版具备拖拽排序
- 不要求复杂模板管理面板
- 不要求高级筛选和搜索体验

---

## 5. MVP 数据边界

本期必须严格区分以下对象：

### 5.1 TaskTemplate
模板资产，用于长期复用团队结构。

### 5.2 TaskTemplateDraft
模板复制后的创建前临时态，可编辑，但尚未创建 task。

### 5.3 TaskCreateRequest
最终运行时创建输入，必须满足所有 role 配置完整。

### 5.4 Task / TaskRole
真正进入运行时的任务与角色实体。

**固定原则**：
- Template 不是运行时实体
- Draft 不是持久化 task
- Task 创建成功后与模板脱钩
- 不允许模板直接跳过 draft 进入运行时

---

## 6. 推荐实施顺序

### Phase 1：先打通“从系统模板创建 Task”闭环

这是最高优先级，也是本期 MVP 的第一阶段。

**包含**：
1. 模板数据结构定义
2. 系统模板 loader
3. 模板列表与预览
4. Template → Task Draft
5. 创建前确认页
6. provider/model 缺失阻断
7. 接入现有 `create_task`

**为什么先做这部分**：
- 用户价值最直接
- 范围最清晰
- 依赖最少
- 能最早验证模板机制是否成立

---

### Phase 2：再做“保存当前 Task 为用户模板”

在 Phase 1 跑通后，再补用户模板沉淀能力。

**包含**：
1. 从当前 task 提取团队结构
2. 保存用户模板
3. 用户模板列表接入创建入口
4. 用户模板复用创建 task

**为什么第二步做这部分**：
- 复用 Phase 1 的模板结构与创建链路
- 先验证“创建链路”，再补“资产沉淀链路”更稳妥

---

## 7. 建议的首批系统模板

首版不宜铺太多模板，建议只放最常见的几个：

1. **软件开发团队**
   - Product Manager
   - Backend Engineer
   - Frontend Engineer
   - QA Engineer

2. **代码修复与验证团队**
   - Developer
   - QA Engineer

如首版容量允许，可追加：

3. **功能交付小队**
   - Product Manager
   - Full-stack Developer
   - Designer
   - QA Engineer

4. **需求分析与实现团队**
   - Product Manager
   - System Architect
   - Developer

## 8. 关键产品原则

### 8.1 Template 复用结构，Archetype 定义行为
- Template 负责角色组合复用
- Archetype 负责单个角色行为基线
- Template 不替代 Archetype

### 8.2 模板可以不完整，Task 不可以
- 模板中的 provider/model 可以为空
- 真正创建 task 前，每个 role 必须补齐 provider/model
- 因此创建前确认是必经步骤

### 8.3 模板与 Task 脱钩
- 从模板创建出来的 task 与模板无运行时绑定
- 修改模板不影响既有 task
- 修改 task 不自动回写模板

### 8.4 用户保留最终控制权
- 模板只提供默认结构和建议值
- 用户始终在创建前完成最终确认
- 系统不能绕过确认流程直接创建

---

## 9. MVP 验收标准

只有当以下条件全部满足时，才视为 Team Templates MVP 完成：

1. 用户能在新建 task 时选择系统模板
2. 用户能查看模板预览
3. 选择模板后，系统能生成 task draft
4. 用户能在创建前修改 task 名称、role 名称、archetype、provider/model
5. 任一 role 缺少 provider/model 时，系统阻止创建
6. 配置完整后，系统能成功创建 task 并进入现有运行时流程
7. 用户能将当前 task 保存为用户模板
8. 用户模板能在后续新建 task 时再次使用

---

## 10. 工作交接建议

### 10.1 交给后端开发的重点
- 建立 `TaskTemplate` / `TemplateRole` / `TaskTemplateDraft` 对应的数据结构
- 实现系统模板读取能力
- 实现用户模板保存与读取接口
- 保持模板与运行时 task 的边界清晰
- 严格执行 provider/model 完整性校验

### 10.2 交给前端开发的重点
- 在 Task 创建入口增加“空白创建 / 从模板创建”分支
- 完成模板列表与预览 UI
- 完成 Template → Draft 的编辑体验
- 完成创建前确认页
- 明确缺失 provider/model 的阻断提示

### 10.3 交给 QA 的重点
- 验证系统模板和用户模板都能走完整创建链路
- 验证缺少 provider/model 时无法误创建
- 验证保存模板后可再次复用
- 验证模板与 task 修改后互不影响

---

## 11. 一句话结论

> **先解决“快创建”，再解决“可沉淀”；先打通系统模板创建链路，再补用户模板保存链路。**
