# 任务成员动态管理需求文档

**文档版本**: v1.0  
**创建日期**: 2026-05-01  
**状态**: 待实施

---

## 1. 需求背景

### 1.1 当前限制

当前系统在创建任务时需要一次性确定所有角色成员,任务创建后无法动态调整团队组成。这导致:

1. **测试场景受限**: 在测试 handoff 流程时,如果发现缺少某个角色,必须重新创建任务
2. **真实场景不灵活**: 用户在任务执行过程中可能发现需要增加新角色(如 QA)或删除不需要的角色
3. **试错成本高**: 用户必须在创建任务前就确定完整的团队配置,无法根据实际情况动态调整

### 1.2 触发场景

**当前遇到的问题**:
- 测试 handoff 流程时,任务中缺少 Backend Developer 角色
- 日志显示 `handoff_candidates=none`,因为推荐的角色不在任务中
- 必须重新创建任务才能继续测试

**典型使用场景**:
1. 任务执行中发现需要增加专家角色(如安全审计、性能优化)
2. 某个角色的工作已完成,希望从团队中移除
3. 调整角色顺序以优化协作流程
4. 临时添加角色进行特定阶段的工作

## 2. 功能需求

### 2.1 核心功能

#### 2.1.1 添加角色成员

**功能描述**: 在任务运行时动态添加新的角色成员

**输入参数**:
```typescript
interface AddRoleRequest {
  task_id: string;
  role: RoleConfig;  // 与创建任务时的 RoleConfig 结构相同
}
```

**业务规则**:
- 角色名称在任务内必须唯一
- 新角色的 `display_order` 可以指定,或自动设置为当前最大值 + 1
- 新角色会自动创建一个空的会话(session)
- 添加后立即可用于 handoff 推荐

**返回结果**:
- 成功: 返回新创建的角色信息(包含 `role_id`)
- 失败: 返回错误信息(如角色名称重复)

#### 2.1.2 删除角色成员

**功能描述**: 从任务中删除指定的角色成员

**输入参数**:
```typescript
interface RemoveRoleRequest {
  task_id: string;
  role_id: string;
}
```

**业务规则**:
- 不能删除当前激活的角色(必须先切换到其他角色)
- 删除角色时,该角色的会话历史**保留但标记为已删除**
- 删除后,其他角色的 handoff 推荐列表会自动更新
- 至少保留一个角色(不能删除最后一个角色)

**返回结果**:
- 成功: 返回删除确认信息
- 失败: 返回错误信息(如角色不存在、角色正在使用中)

#### 2.1.3 更新角色配置

**功能描述**: 更新现有角色的配置(如顺序、handoff 开关)

**输入参数**:
```typescript
interface UpdateRoleRequest {
  task_id: string;
  role_id: string;
  updates: {
    display_order?: number;
    handoff_enabled?: boolean;
    name?: string;  // 可选:重命名角色
  };
}
```

**业务规则**:
- 更新 `display_order` 时,自动调整其他角色的顺序以避免冲突
- 重命名角色时,新名称必须在任务内唯一
- 不能修改角色的 `archetype_id`(这会改变角色的本质)

**返回结果**:
- 成功: 返回更新后的角色信息
- 失败: 返回错误信息

### 2.2 UI 交互

#### 2.2.1 添加角色入口

**位置**: 任务详情页 / Team Brief 界面

**交互流程**:
1. 用户点击"添加成员"按钮
2. 弹出角色配置对话框(与创建任务时的角色配置相同)
3. 用户填写角色信息并确认
4. 系统创建角色并刷新团队列表

#### 2.2.2 删除角色入口

**位置**: 角色卡片的操作菜单

**交互流程**:
1. 用户在角色卡片上点击"删除"按钮
2. 系统弹出确认对话框,提示:
   - 该角色的会话历史将被保留但不可访问
   - 如果有其他角色推荐交接给该角色,推荐列表会更新
3. 用户确认后删除

**特殊情况**:
- 如果尝试删除当前激活的角色,提示"请先切换到其他角色"
- 如果尝试删除最后一个角色,提示"任务至少需要一个角色"

#### 2.2.3 调整角色顺序

**位置**: Team Brief 界面

**交互方式**:
- 拖拽排序(drag & drop)
- 或者在角色配置中手动修改 `display_order`

## 3. 技术实现

### 3.1 API 设计

#### 3.1.1 扩展 TaskUpdateRequest

```rust
#[derive(Deserialize)]
pub struct TaskUpdateRequest {
    // 现有字段
    pub name: Option<String>,
    pub description: Option<String>,
    pub icon: Option<String>,
    
    // 新增字段
    pub add_roles: Option<Vec<RoleConfig>>,
    pub remove_role_ids: Option<Vec<String>>,
    pub update_roles: Option<Vec<RoleUpdateRequest>>,
}

#[derive(Deserialize)]
pub struct RoleUpdateRequest {
    pub role_id: String,
    pub display_order: Option<i32>,
    pub handoff_enabled: Option<bool>,
    pub name: Option<String>,
}
```

#### 3.1.2 新增独立 API

或者,为了更清晰的职责分离,可以创建独立的 API:

```rust
// 添加角色
#[tauri::command]
async fn add_task_role(
    task_id: String,
    role: RoleConfig
) -> Result<TaskRole, String>

// 删除角色
#[tauri::command]
async fn remove_task_role(
    task_id: String,
    role_id: String
) -> Result<(), String>

// 更新角色
#[tauri::command]
async fn update_task_role(
    task_id: String,
    role_id: String,
    updates: RoleUpdateRequest
) -> Result<TaskRole, String>
```

**推荐**: 使用独立 API,职责更清晰,错误处理更精确

### 3.2 数据库变更

#### 3.2.1 角色表(roles)

**新增字段**:
```sql
ALTER TABLE roles ADD COLUMN deleted_at TEXT DEFAULT NULL;
```

**说明**:
- `deleted_at`: 软删除标记,记录删除时间
- 查询时默认过滤 `deleted_at IS NULL` 的记录
- 保留已删除角色的会话历史,用于审计和回溯

#### 3.2.2 会话表(sessions)

**无需变更**,会话通过 `role_id` 关联角色,角色软删除后会话仍然保留

### 3.3 Handoff 推荐更新

**影响点**: `build_role_prompt_contexts` 函数

**变更逻辑**:
1. 添加角色后,重新计算所有角色的 `recommended_handoff_roles`
2. 删除角色后,从其他角色的推荐列表中移除该角色
3. 更新角色顺序后,重新排序 `role_roster`

**实现方式**:
- 在 `add_task_role` / `remove_task_role` 后,触发 `rebuild_role_contexts`
- 或者在每次获取任务信息时动态计算(当前实现方式)

### 3.4 前端状态管理

**影响组件**:
- `TaskDetail.tsx`: 任务详情页
- `TeamBrief.tsx`: 团队概览
- `RoleCard.tsx`: 角色卡片

**状态更新**:
- 添加/删除角色后,刷新任务信息
- 更新角色后,局部更新角色卡片
- 使用乐观更新提升用户体验

## 4. 边界情况处理

### 4.1 并发操作

**场景**: 用户在多个窗口同时操作同一个任务

**处理方式**:
- 使用数据库事务保证原子性
- 前端定期轮询任务状态,检测变更
- 或使用 WebSocket 实时推送变更

### 4.2 角色删除后的 Handoff

**场景**: 角色 A 推荐交接给角色 B,但角色 B 被删除了

**处理方式**:
- 后端在解析 handoff 时,检查目标角色是否存在
- 如果不存在,返回 `handoff_target_unresolved` 错误
- 前端提示用户"目标角色已被删除,请选择其他角色"

### 4.3 最后一个角色

**场景**: 用户尝试删除任务中的最后一个角色

**处理方式**:
- 后端检查角色数量,如果只剩一个则拒绝删除
- 返回错误信息: "任务至少需要一个角色"

### 4.4 当前激活角色

**场景**: 用户尝试删除当前正在使用的角色

**处理方式**:
- 后端检查 `active_role_id`,如果是当前角色则拒绝删除
- 返回错误信息: "无法删除当前激活的角色,请先切换到其他角色"

## 5. 验收标准

### 5.1 功能完整性

- ✅ 可以在任务运行时添加新角色
- ✅ 可以删除不需要的角色(非当前激活角色)
- ✅ 可以更新角色的 `display_order` 和 `handoff_enabled`
- ✅ 添加角色后,该角色立即出现在 handoff 推荐列表中
- ✅ 删除角色后,其他角色的推荐列表自动更新

### 5.2 数据一致性

- ✅ 角色名称在任务内唯一
- ✅ 删除角色后,会话历史保留但不可访问
- ✅ 至少保留一个角色
- ✅ 不能删除当前激活的角色

### 5.3 用户体验

- ✅ 添加角色的对话框与创建任务时一致
- ✅ 删除角色前有明确的确认提示
- ✅ 操作后界面立即更新,无需手动刷新
- ✅ 错误信息清晰易懂

### 5.4 性能要求

- ✅ 添加/删除角色的响应时间 < 500ms
- ✅ 更新角色顺序的响应时间 < 200ms
- ✅ 不影响现有任务的加载性能

## 7. 风险与依赖

### 7.1 技术风险

**风险**: 删除角色后,历史会话数据可能无法正确显示

**缓解措施**:
- 使用软删除而非硬删除
- 在 UI 中明确标记已删除角色的会话
- 提供"查看已删除角色"的选项(可选)

### 7.2 用户体验风险

**风险**: 用户误删角色导致数据丢失

**缓解措施**:
- 删除前明确提示后果
- 提供"撤销删除"功能(可选,Phase 4)
- 保留会话历史,允许恢复角色(可选,Phase 4)

### 7.3 依赖项

**前置条件**:
- 当前的任务管理系统稳定运行
- Handoff 功能已修复并测试通过

**后续功能**:
- 角色模板管理(快速添加常用角色组合)
- 角色权限管理(限制某些角色的操作)

## 8. 未来扩展

### 8.1 角色模板

**功能**: 预定义常用的角色组合,一键添加

**示例**:
- "软件交付团队": PM + Backend + Frontend + QA
- "设计评审团队": Designer + PM + Frontend
- "安全审计团队": Security Engineer + Backend + DevOps

### 8.2 角色权限

**功能**: 限制某些角色的操作权限

**示例**:
- QA 角色只能查看代码,不能修改
- PM 角色可以添加/删除其他角色
- 普通角色不能修改任务配置

### 8.3 角色恢复

**功能**: 恢复已删除的角色及其会话历史

**实现**:
- 在"已删除角色"列表中选择角色
- 点击"恢复"按钮
- 角色重新出现在团队中,会话历史可访问
