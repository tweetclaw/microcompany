## 2. 背景与问题定义

在 Team Templates 引入之前，用户每次创建多角色 task 时都要重复完成类似配置：

- 手动添加角色
- 为每个角色选择 archetype / identity
- 为每个角色配置 provider / model
- 反复搭建相似的团队结构

这带来两个核心问题：

1. **创建效率低**：高频用户每次建 task 都重复劳动
2. **有效团队无法沉淀**：一次成功的团队结构不能复用为长期资产

Team Templates 的目标是把一次有效的团队配置沉淀为模板，并允许用户在后续新建 task 时直接复用，同时在进入运行时前完成必要校验与补全。

## 3. 目标演进：从 MVP 到当前实现

### 3.1 MVP 初始目标

最初 MVP 聚焦两个最高价值场景：

1. **从模板快速创建 task**
2. **将当前 task 的团队结构保存为模板**

其硬性约束包括：

- 模板不能直接变成运行时 task，必须先进入 draft / 确认流程
- 创建前必须检查所有 role 的 provider / model 是否完整
- 模板和 task 必须解耦，后续互不回写

### 3.2 后续范围扩展

在 MVP 打通后，需求自然扩展到两个方向：

1. **模板侧能力增强**
   - 用户模板编辑
   - 模板内容管理
   - 模板作为长期配置资产管理

2. **运行时团队管理增强**
   - 运行中 task 动态添加角色
   - 编辑角色配置
   - 删除角色
   - 拖拽排序角色

当前代码和测试结果表明，上述两类能力已经形成可用闭环，但更高阶的 Phase 3 规划项（如 handoff 可视化、批量操作、角色恢复）仍未完成。

## 4. 核心产品原则

### 4.1 Template 复用结构，Archetype / Identity 定义行为

- Template 负责沉淀和复用团队结构
- Archetype / Identity 负责单个角色行为基线
- 模板不替代角色行为定义

### 4.2 模板可以不完整，运行时 task 不可以

- 模板中的 provider / model 可以为空或仅为建议值
- 真正创建 task 前，每个 role 必须补齐 provider / model
- 创建前确认是必经步骤

### 4.3 模板与 Task 解耦

- 从模板创建出来的 task 与模板无运行时绑定
- 修改模板不影响既有 task
- 修改 task 不自动回写模板

### 4.4 用户保留最终控制权

- 模板只提供默认结构和建议值
- 用户在创建前完成最终确认
- 系统不能绕过确认流程直接创建运行时 task

## 5. 数据边界与对象模型

本功能涉及四类核心对象：

### 5.1 TaskTemplate
长期复用的模板资产，保存团队结构定义。

### 5.2 TemplateRole / TaskTemplateRole
模板内角色定义，包含：

- 角色顺序
- 角色名称
- archetype / identity
- provider / model（可选）

### 5.3 TaskTemplateDraft
从模板复制后的创建前临时态，可在前端编辑，但不是持久化运行时 task。

### 5.4 Task / TaskRole
真正进入运行时的任务与角色实体。

**固定原则**：

- Template 不是运行时实体
- Draft 不是持久化 task
- Task 创建成功后与模板脱钩
- 不允许模板绕过确认直接进入运行时

## 7. 现有 UI / 交互结构

### 7.1 Task 创建入口

新建 task 时，用户进入模板选择器：

- 可选择 Blank Task
- 可选择系统模板
- 可选择用户模板
- 可查看模板详情
- 可继续进入 draft 编辑 / 创建确认

### 7.2 模板管理入口

基于后续产品调整，模板管理不再只埋在“新建 task”流程内部，而是在 **task 列表顶部** 提供独立入口，原因包括：

- 模板本质上是 task 团队结构资产，适合在 task 主入口附近管理
- 设置页是 modal，空间受限，不适合承载复杂模板管理 UI
- 用户可以在同一操作区完成：新建 task、查看 task、管理模板

### 7.3 模板编辑弹窗行为

模板编辑弹窗已调整为：

- 只有点击明确的关闭按钮 / Cancel / Save 后才消失
- 点击外部遮罩不会关闭

这样可避免误触导致编辑内容丢失。

## 11. 主要设计取舍

### 11.1 为什么模板不能直接创建运行时 task

因为模板是结构资产，不是最终运行时配置。若允许直接创建，会导致：

- provider / model 缺失时误创建
- 用户失去最终确认机会
- 模板与运行时边界混乱

所以必须保留 draft / 确认层。

### 11.2 为什么模板管理入口不放在 Settings

虽然从信息架构角度 Settings 能承载“管理类”内容，但当前产品中 Settings 是 modal 弹层，存在两个问题：

- 空间受限，不适合复杂模板管理界面
- 模板与 task 工作流关系更紧密，放在 task 入口附近更符合心智

因此最终采用 task 列表顶部的独立入口，是更符合操作频率和界面空间的方案。

### 11.3 为什么编辑模板弹窗不能点外部关闭

模板编辑属于高输入密度场景，误触关闭会带来明显损失。因此改为“仅显式关闭”，是以数据安全优先于轻量交互。

### 13.1 测试范围

本轮测试覆盖：

- 运行时角色新增 / 编辑 / 删除 / 排序
- 应用重启后的持久化

### 13.2 准备工作

1. 启动开发服务器：`npm run dev`
2. 启动 Tauri 桌面端
3. 打开开发者控制台（查看日志和错误）
4. 创建一个至少含 2 个角色的 task

### 13.3 测试项

#### B1: 运行中任务 — 添加角色

**操作步骤**:
1. 打开一个已有的 Task，进入角色列表视图
2. 点击「+ 添加角色」
3. 填写角色名称（如 `Reviewer`）
4. 选择身份类型（identity）
5. 选择 Provider（如 `openai`）
6. 点击确认

**预期效果**:
- 提示成功添加角色
- 角色列表中出现新角色
- 新角色的 provider 显示正确
- 控制台无报错

**通过标准**: 新角色出现在列表中，provider 信息正确。

#### B2: 运行中任务 — 编辑角色

**操作步骤**:
1. 点击某个角色的编辑图标
2. 修改 name
3. 修改 provider
4. 点击确认

**预期效果**:
- 提示更新成功
- 角色卡片即时刷新
- 控制台无报错

**通过标准**: 角色卡片显示修改后的值，无需手动刷新页面。

#### B3: 运行中任务 — 删除角色

**操作步骤**:
1. 点击某个角色的删除图标
2. 确认删除对话框弹出
3. 点击确认

**预期效果**:
- 提示成功删除角色
- 角色列表中该角色消失
- 控制台无报错

**通过标准**: 角色被移除，列表即时更新。

#### B4: 运行中任务 — 拖拽排序角色

**操作步骤**:
1. 在至少 3 个角色的列表中按住某个角色卡片拖动
2. 拖到新位置后松开

**预期效果**:
- 角色顺序变化
- 新顺序被保存
- 重新打开 task 后顺序不变
- 控制台无报错

**通过标准**: 拖拽后顺序保持，重开 task 后顺序不变。

#### B5: 重启后角色数据持久化

**操作步骤**:
1. 执行 B1 或 B2
2. 完全退出应用
3. 重新启动应用
4. 打开同一个 Task

**预期效果**:
- 之前添加/编辑的角色仍存在
- provider / name 等字段保持一致
- 排序保持不变

**通过标准**: 数据在应用重启后完全保留。

#### B6: 回归 — 模板编辑 + provider 保存

**操作步骤**:
1. 进入模板列表
2. 点击某个模板的 Edit
3. 修改某个角色的 provider
4. 保存
5. 退出编辑后重新打开该模板

**预期效果**:
- 保存成功，无报错
- 再次打开后修改仍保留
- 卡片状态正确刷新

**通过标准**: 模板编辑功能仍正常工作。

#### B7: 回归 — Save Task as Template

**操作步骤**:
1. 打开一个有角色的 Task
2. 点击「保存为模板」
3. 输入模板名称和描述
4. 保存后回到模板列表

**预期效果**:
- 新模板出现在用户模板列表中
- 新模板可以打开详情
- 角色 provider 信息正确保留

**通过标准**: 另存为模板功能正常工作。
