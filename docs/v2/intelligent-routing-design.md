# 智能路由方案设计

**文档状态**: 已确定  
**创建日期**: 2026-05-03  
**最后更新**: 2026-05-03  
**目标**: 设计一个智能机制,在 AI 工作结束后自动判断是否需要交接给下一个角色

---

## 问题背景

当前系统中,一个 AI 角色完成工作后,无法智能判断:
- 当前工作是否已完成,可以继续由同一角色处理?
- 还是需要交接给下一个角色继续处理?

需要设计一个**智能路由机制**来解决这个问题。

---

## 最终方案: AI 推荐 + 用户确认 (混合方案)

### 核心思路

结合方案 1 和方案 2 的优势,采用**混合方案**:
- 使用独立的 JSON API 调用来**提取和结构化** handoff 内容
- AI 给出**推荐的接手角色**
- 用户通过 UI **手动确认**最终选择
- 在 UI 上**显示提取的任务摘要** (task_summary)

### 工作流程

```
AI 对话结束
    ↓
调用独立 JSON API (轻量级提取)
    ↓
返回结构化 handoff 信息
    ↓
UI 显示:
  - 任务摘要 (task_summary)
  - 推荐角色高亮
  - 所有角色的【接手】按钮亮起
    ↓
用户点击某个角色的【接手】按钮
    ↓
将结构化信息传递给被选中的角色
```

### 技术设计

#### JSON API 调用

**目的**: 提取和结构化 handoff 内容,而非做路由判断

**输入结构**:
```json
{
  "role": "产品经理",
  "last_message": "需求分析完成。核心功能包括:\n1. 用户登录\n2. 数据展示\n3. 报表导出\n\n建议交接给开发工程师实现以上功能。"
}
```

**输出结构**:
```json
{
  "has_handoff": true,
  "task_summary": "实现用户登录、数据展示和报表导出功能",
  "key_requirements": [
    "用户登录",
    "数据展示",
    "报表导出"
  ],
  "suggested_role": "开发工程师"
}
```

**字段说明**:
- `has_handoff`: 是否检测到 handoff 意图
- `task_summary`: 任务摘要 (在 UI 上显示)
- `key_requirements`: 关键需求列表
- `suggested_role`: AI 推荐的接手角色 (仅作为 UI 提示)

#### UI 设计

```
┌─────────────────────────────────────────────────┐
│ 产品经理 (当前对话)                                │
│ [对话内容...]                                      │
│                                                   │
│ > 需求分析完成,建议交接给开发工程师                  │
│                                                   │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│ 📋 任务摘要                                       │
│ 实现用户登录、数据展示和报表导出功能                │
│                                                   │
│ 关键需求:                                         │
│ • 用户登录                                        │
│ • 数据展示                                        │
│ • 报表导出                                        │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│ 选择接手成员                                      │
│                                                   │
│ ⭐ 开发工程师  [接手] ← AI 推荐 (高亮显示)         │
│    测试工程师  [接手]                             │
│    设计师      [接手]                             │
│                                                   │
└─────────────────────────────────────────────────┘
```

**UI 交互逻辑**:
1. AI 对话结束后,系统自动调用 JSON API
2. 如果 `has_handoff: true`:
   - 显示任务摘要卡片
   - 所有其他角色的【接手】按钮亮起
   - 如果有 `suggested_role`,该角色高亮显示 (⭐ 标记)
3. 用户点击任意角色的【接手】按钮
4. 将 `task_summary` 和 `key_requirements` 传递给被选中的角色

#### 系统提示词设计

```
你是一个 AI 团队协作助手。分析当前对话的最后一条消息,判断是否包含任务交接意图。

任务:
1. 判断是否需要交接给其他角色 (has_handoff: true/false)
2. 如果需要交接,提取任务摘要 (task_summary)
3. 提取关键需求列表 (key_requirements)
4. 推荐最合适的接手角色 (suggested_role)

角色职责参考:
- 产品经理: 需求分析、功能规划
- 开发工程师: 代码实现、技术方案
- 测试工程师: 测试用例、质量保证
- 设计师: UI/UX 设计、视觉规范

返回 JSON 格式:
{
  "has_handoff": boolean,
  "task_summary": string,
  "key_requirements": string[],
  "suggested_role": string
}

注意:
- task_summary 应简洁明了,一句话概括任务
- key_requirements 提取核心要点,不超过 5 条
- suggested_role 必须是上述角色之一
- 如果没有明确的交接意图,返回 has_handoff: false
```

### 方案优势

1. **可控性** - 用户完全掌控最终决策,避免 AI 误判
2. **智能辅助** - AI 推荐减少用户思考负担
3. **结构化准确** - JSON API 保证提取准确性
4. **用户体验好** - 显示任务摘要,用户一目了然
5. **成本可控** - 只用于轻量级提取,不做复杂推理
6. **可扩展** - 未来可以增加"自动接手"开关

### 与旧方案的关系

**代码清理策略**:
- **保留** handoff 内容生成逻辑 (因为内容已经很好)
- **删除** 占位符替换的错误实现 (`{{role}}` 相关代码)
- **新增** JSON API 调用逻辑
- **新增** UI 接手按钮和任务摘要显示

参考 [清理文档](./cleanup-invalid-handoff-solution.md) 了解详细清理步骤。

## 备选方案 (供参考)

以下是讨论过程中考虑的其他方案,记录在此供未来参考。

### 备选方案 A: 完全自动化的裁判 AI

**核心思路**: 使用独立的 JSON API 调用来**判断路由决策**并**自动执行**,无需用户确认。

**优势**:
- 完全自动化,用户无需干预
- 工作流自动流转,效率最高

**劣势**:
- 可能出现误判,导致工作流混乱
- 用户失去控制权
- 调试困难

**为什么没有选择**: 第一阶段优先可控性,等测试效果好之后再探索自动化。

---

### 备选方案 B: 纯手动接手 (无 AI 辅助)

**核心思路**: 只提供【接手】按钮,不使用任何 AI 提取或推荐。

**优势**:
- 实施最简单
- 无额外 API 成本
- 用户完全可控

**劣势**:
- 无法准确提取 handoff 内容 (当前代码无法弹出对话框的根本原因)
- 用户需要手动复制粘贴任务描述
- 无智能推荐,用户负担重

**为什么没有选择**: 对当前的结构化方案不自信,需要 JSON API 保证提取准确性。

---

### 备选方案 C: 正则表达式提取

**核心思路**: 使用正则表达式从 AI 输出中提取 handoff 内容,不调用额外的 API。

**优势**:
- 无额外 API 成本
- 实施相对简单

**劣势**:
- 提取准确性依赖 AI 输出格式的一致性
- 不同角色的输出格式可能不一致
- 难以处理复杂的自然语言表达
- 维护成本高 (需要不断调整正则规则)

**为什么没有选择**: 提取准确性无法保证,JSON API 是更可靠的方案。

### 方案 1: 裁判 AI (独立 API 调用)

#### 核心思路

为 AI 团队增加一个**裁判 AI**,专门负责判断工作流转:
- **不走 claurst 子模块**
- 直接通过 API key 发起独立的 AI 大模型请求
- 可以设置**系统提示词**
- 返回**结构化 JSON 格式**,而非聊天格式

#### 技术优势

1. **可控性强**
   - 可以精确设置系统提示词,定义裁判 AI 的判断逻辑
   - 返回格式可控,便于系统解析和调度

2. **自主调度**
   - 系统能识别 JSON 结果并自动执行调度决策
   - 无需人工干预,实现真正的智能路由

3. **独立性**
   - 不依赖 claurst 子模块的限制
   - 可以使用不同的模型或参数配置

#### 实施要点

**输入结构**:
```json
{
  "current_role": "产品经理",
  "conversation_history": "...",
  "last_message": "...",
  "available_roles": ["开发工程师", "测试工程师", "设计师"]
}
```

**输出结构**:
```json
{
  "decision": "handoff" | "continue",
  "target_role": "开发工程师",
  "reason": "当前任务需要技术实现,建议交接给开发工程师",
  "handoff_context": {
    "task_summary": "...",
    "key_requirements": ["...", "..."],
    "expected_output": "..."
  }
}
```

**系统提示词示例**:
```
你是一个 AI 团队的工作流协调员。根据当前对话内容,判断:
1. 当前角色是否已完成其职责范围内的工作
2. 是否需要交接给其他角色
3. 如果需要交接,应该交给哪个角色

判断标准:
- 产品经理完成需求分析后 → 交接给开发工程师
- 开发工程师完成代码实现后 → 交接给测试工程师
- 测试工程师发现 bug 后 → 交接回开发工程师
- 所有角色完成各自工作后 → 标记为 continue (等待新任务)

返回 JSON 格式,包含 decision, target_role, reason, handoff_context 字段。
```

#### 潜在问题

1. **额外 API 成本** - 每次判断都需要调用一次独立的 AI API
2. **延迟增加** - 需要等待裁判 AI 的响应才能继续
3. **判断准确性** - 依赖裁判 AI 的理解能力,可能出现误判
4. **维护复杂度** - 需要维护裁判 AI 的提示词和判断逻辑

---

### 方案 2: 手动接手按钮 (用户选择)

#### 核心思路

在 UI 层面提供**手动接手机制**:
- 每个角色下方有一个**【接手】按钮**
- 当前 AI 对话结束后,其他成员的【接手】按钮**亮起**
- 用户选中某个角色,将前一个 AI 的任务描述传递给被选中的角色

#### 技术优势

1. **用户可控**
   - 用户完全掌控工作流转,避免 AI 误判
   - 灵活性高,可以根据实际情况调整

2. **实施简单**
   - 不需要额外的 AI API 调用
   - UI 改动相对较小

3. **成本低**
   - 无额外 API 成本
   - 无延迟问题

#### 实施要点

**UI 设计**:
```
┌─────────────────────────────────────┐
│ 产品经理 (当前对话)                    │
│ [对话内容...]                          │
│                                       │
│ > 需求分析完成,建议交接给开发工程师      │
│                                       │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ 其他团队成员                           │
│                                       │
│ 开发工程师  [接手] ← 按钮亮起           │
│ 测试工程师  [接手] ← 按钮亮起           │
│ 设计师      [接手] ← 按钮亮起           │
│                                       │
└─────────────────────────────────────┘
```

**数据提取逻辑**:
- 需要从当前 AI 的输出中**提取 handoff 内容**
- 要求输出有一定**结构化**,才能准确分离任务描述
- 可能需要使用正则表达式或简单的文本解析

**提取示例**:
```
原始输出:
"需求分析完成。核心功能包括:
1. 用户登录
2. 数据展示
3. 报表导出

建议交接给开发工程师实现以上功能。"

提取结果:
{
  "handoff_detected": true,
  "target_role_hint": "开发工程师",
  "task_description": "实现以下功能:\n1. 用户登录\n2. 数据展示\n3. 报表导出"
}
```

#### 潜在问题

1. **用户负担** - 需要用户手动判断和点击,增加操作步骤
2. **提取准确性** - 如果 AI 输出格式不规范,可能无法准确提取 handoff 内容
3. **一致性问题** - 不同 AI 角色的输出格式可能不一致,增加提取难度
4. **扩展性** - 如果团队成员很多,按钮列表会很长

#### 与旧方案的关系

**重要**: 如果选择这个方案,**第一阶段的清理可能不需要那么彻底**,因为:
- 旧方案确实能产生一段 handoff 内容
- 可以复用旧方案中的内容生成逻辑
- 只需要删除占位符替换的错误实现
- 保留结构化输出相关代码

## 方案对比总结

| 维度 | 方案 1: 裁判 AI | 方案 2: 手动接手按钮 |
|------|----------------|---------------------|
| **自动化程度** | 高 (完全自动) | 低 (需要用户操作) |
| **准确性** | 依赖 AI 判断 | 依赖用户判断 |
| **实施复杂度** | 高 | 中 |
| **API 成本** | 有额外成本 | 无额外成本 |
| **延迟** | 有额外延迟 | 无额外延迟 |
| **用户体验** | 无感知 (自动) | 需要手动操作 |
| **灵活性** | 低 (依赖提示词) | 高 (用户可控) |
| **与旧方案关系** | 需要完全清理 | 可以部分复用 |
