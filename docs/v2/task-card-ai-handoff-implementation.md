# Task 测试卡：Team Templates MVP

**用途**：给 Task AI 下发一个真实任务，用来观察现有 task 协作与 handoff 是否自然发生。

---

## 任务描述（给 PM 的消息）

我们需要实现 Team Templates 功能，让用户能把一组可复用的 task 团队结构保存为模板，并在创建新 task 时直接从模板开始。

当前用户每次创建多角色 task 时都要手动添加角色、选择 Archetype、配置 provider/model，重复劳动严重。Team Templates 要解决的核心问题是：**把一次有效的任务团队配置，变成之后可反复复用的组织能力。**

核心需求：
1. 用户可以从模板创建 task（选择系统模板或用户模板）
2. 用户可以把当前 task 的团队结构保存为模板
3. 模板包含：角色顺序、角色名称、推荐 archetype、可选的 provider/model
4. 创建前必须有确认界面，补齐所有 role 的 provider/model

请先收敛 MVP 范围：
- 最小功能范围是什么
- 应该先做哪一部分

参考文档：`docs/v2/team-templates-user-guide.md`

---

## 推荐角色

PM、Backend、Frontend、QA。
## 3. 推荐协作顺序

按这个顺序测试最清楚：

1. **PM → Backend**
2. **Backend → Frontend**
3. **Frontend → QA**

不要一开始就要求所有角色完整实现；先让每个角色处理自己当前阶段最该处理的部分，再看系统是否把工作自然推进下去。

---

## 4. 你在测试时应该观察什么

只看这几件事：

- 当前角色是否理解了自己负责的部分
- 当前角色是否能自然指出下一步该谁接手
- 系统是否出现转交建议 / 转交确认
- 下一角色接手后，是否真的基于上一个角色的结果继续工作

---

## 5. 成功标准

满足下面几点，就说明这张 task 卡足够适合测试：

- 任务下发方式简单明确
- 角色分工清楚
- 协作顺序自然
- 不需要用户知道任何内部 handoff 标记或特殊语法
- 用户只像真实地给团队派发一个开发任务

---

## 6. 失败信号

如果出现下面情况，说明这张 task 卡还不够好：

- 任务说明太长，AI 抓不住重点
- 角色设置不清楚，谁都能做、谁都不像该做
- 用户必须知道内部 handoff 写法，才能让流程跑起来
- 下一角色接手时，像重新开始，而不是延续上一步结果
