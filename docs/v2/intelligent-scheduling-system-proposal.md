# 智能调度系统方案

**文档版本**: v1.0  
**创建日期**: 2026-04-22  
**项目**: MicroCompany

## 1. 需求概述

### 1.1 业务目标

从手动调度消息和任务转变为智能自动调度，让系统能够自动识别消息应该发送给哪个角色，自动执行任务流程，支持条件分支和循环，实现真正的多角色协作自动化。

### 1.2 当前痛点

**手动调度的问题**：
- 用户需要手动选择目标角色转发消息
- 无法定义工作流程和执行顺序
- 无法实现条件判断（如：测试失败返回编码阶段）
- 无法并行执行多个角色任务
- 缺乏执行状态监控和日志

**用户期望**：
- 定义一次流程，自动执行
- 系统智能判断下一步应该做什么
- 可视化监控执行进度
- 支持复杂的业务逻辑（分支、循环、并行）

### 1.3 核心功能

1. **工作流定义**
   - 可视化流程图编辑器
   - 支持多种节点类型（角色、条件、并行、循环）
   - 定义节点间的连接和路由规则

2. **自动调度执行**
   - 按照工作流定义自动执行
   - 消息自动路由到对应角色
   - 等待角色回复后继续下一步

3. **智能路由**
   - 基于规则的消息路由
   - 基于关键词的条件判断
   - 可选：基于 AI 的意图识别

4. **执行监控**
   - 实时显示执行状态
   - 可视化当前执行位置
   - 详细的执行日志
   - 支持暂停/恢复/取消

## 3. 数据模型设计

### 3.1 核心实体

**工作流 (Workflow)**
- 工作流 ID
- 所属任务 ID
- 工作流名称
- 工作流描述
- 工作流定义（JSON 格式）
- 是否激活
- 创建时间、更新时间

**工作流执行实例 (WorkflowExecution)**
- 执行 ID
- 工作流 ID
- 任务 ID
- 执行状态（运行中、暂停、完成、失败、取消）
- 当前节点 ID
- 执行上下文（变量、中间结果）
- 开始时间、完成时间
- 错误信息

**执行日志 (ExecutionLog)**
- 日志 ID
- 执行 ID
- 节点 ID
- 节点类型
- 角色 ID
- 动作（进入、执行、退出、错误）
- 输入数据
- 输出数据
- 执行耗时
- 时间戳

**路由规则 (RoutingRule)**
- 规则 ID
- 工作流 ID
- 优先级
- 条件定义（JSON 格式）
- 目标角色 ID
- 是否激活
- 创建时间

### 3.2 工作流定义结构

工作流定义采用 JSON 格式存储，包含节点和边两部分：

**节点类型**：
- 角色节点：执行特定角色的任务
- 条件节点：根据条件判断分支
- 并行节点：同时执行多个分支
- 循环节点：重复执行某段流程

**边（连接）**：
- 起始节点
- 目标节点
- 触发条件
- 标签（用于显示）

## 4. 工作流节点类型

### 4.1 角色节点 (Role Node)

**功能**：将消息发送给指定角色，等待角色回复

**配置参数**：
- 角色 ID：指定哪个角色执行
- 自动执行：是否自动发送消息
- 超时时间：等待回复的最长时间
- 重试策略：失败后是否重试
- 最大重试次数

**执行逻辑**：
1. 获取角色的 session_id
2. 构造消息内容（可使用上一节点的输出）
3. 调用消息转发 API
4. 等待角色回复
5. 保存回复到执行上下文
6. 转移到下一个节点

### 4.2 条件节点 (Condition Node)

**功能**：根据条件判断选择不同的分支

**配置参数**：
- 条件类型：关键词、正则、AI 判断
- 条件表达式
- 匹配模式：任意匹配、全部匹配

**条件类型**：
- 关键词匹配：检查消息是否包含特定关键词
- 正则表达式：使用正则匹配消息内容
- AI 意图识别：调用大模型判断消息意图

**执行逻辑**：
1. 获取上一节点的输出
2. 根据条件类型进行评估
3. 返回 true/false
4. 根据结果选择对应的出边

### 4.3 并行节点 (Parallel Node)

**功能**：同时执行多个分支，等待所有分支完成

**配置参数**：
- 分支列表：要并行执行的节点 ID 列表
- 等待策略：全部完成、任意完成、超时

**执行逻辑**：
1. 同时启动所有分支
2. 监控各分支执行状态
3. 根据等待策略决定何时继续
4. 合并各分支的输出

### 4.4 循环节点 (Loop Node)

**功能**：重复执行某段流程，直到满足退出条件

**配置参数**：
- 最大迭代次数：防止死循环
- 退出条件：何时停止循环
- 循环体：要重复执行的节点

**执行逻辑**：
1. 初始化迭代计数器
2. 执行循环体
3. 评估退出条件
4. 如果未满足且未超过最大次数，继续循环
5. 否则退出循环

## 5. 典型工作流场景

### 5.1 软件开发流程

```
开始
  ↓
项目经理（需求分析）
  ↓
架构师（架构设计）
  ↓
条件判断：设计是否通过？
  ├─ 是 → 程序员（编码实现）
  │         ↓
  │       测试员（测试）
  │         ↓
  │       条件判断：测试是否通过？
  │         ├─ 是 → 评审员（代码评审）→ 结束
  │         └─ 否 → 返回程序员（修复 bug）
  │
  └─ 否 → 返回架构师（修改设计）
```

### 5.2 内容创作流程

```
开始
  ↓
策划（确定主题和大纲）
  ↓
撰稿（撰写初稿）
  ↓
编辑（编辑润色）
  ↓
条件判断：是否需要修改？
  ├─ 是 → 返回撰稿（修改）
  └─ 否 → 审核（最终审核）→ 结束
```

### 5.3 数据分析流程

```
开始
  ↓
数据分析师（需求分析）
  ↓
并行执行：
  ├─ 数据工程师（数据清洗）
  └─ 数据工程师（数据建模）
  ↓
合并结果
  ↓
可视化专家（制作图表）
  ↓
结束
```

## 7. 调度引擎设计

### 7.1 状态机模型

**执行状态**：
- 初始化：工作流刚创建
- 运行中：正在执行节点
- 等待中：等待角色回复
- 暂停：用户手动暂停
- 完成：所有节点执行完毕
- 失败：执行过程中出错
- 取消：用户手动取消

**状态转换**：
```
初始化 → 运行中 → 等待中 → 运行中 → ... → 完成
   ↓        ↓        ↓
  取消     暂停     失败
```

### 7.2 消息队列机制

**队列结构**：
- 待执行队列：等待执行的节点
- 执行中队列：正在执行的节点
- 完成队列：已完成的节点

**队列操作**：
- 入队：将节点加入待执行队列
- 出队：从待执行队列取出节点执行
- 优先级：支持节点优先级排序

### 7.3 并发控制

**并发策略**：
- 单个工作流内的并行节点可以并发执行
- 多个工作流可以同时执行
- 限制最大并发数防止资源耗尽

**资源管理**：
- 角色会话池：管理角色的 session
- 消息发送限流：防止过载
- 超时控制：避免长时间等待

### 7.4 错误处理

**错误类型**：
- 节点执行失败：角色返回错误
- 超时：等待回复超时
- 条件评估失败：条件表达式错误
- 系统错误：数据库、网络等错误

**处理策略**：
- 重试：自动重试失败的节点
- 回退：返回上一个节点重新执行
- 跳过：跳过失败节点继续执行
- 终止：停止整个工作流执行

## 8. 智能路由设计

### 8.1 基于规则的路由

**关键词匹配**：
- 定义关键词列表
- 检查消息是否包含关键词
- 支持多个关键词的与/或逻辑

**正则表达式**：
- 使用正则匹配消息内容
- 支持复杂的模式匹配
- 提取匹配的内容作为变量

**示例**：
```
规则 1: 如果消息包含"需求"或"功能" → 路由到项目经理
规则 2: 如果消息包含"bug"或"错误" → 路由到测试员
规则 3: 如果消息包含"设计"或"架构" → 路由到架构师
```

### 8.2 基于 AI 的路由（可选）

**意图识别**：
- 使用大模型分析消息内容
- 识别用户意图（需求分析、编码、测试等）
- 返回最合适的角色

**上下文感知**：
- 考虑对话历史
- 考虑当前执行状态
- 考虑角色的专长

**优势**：
- 更灵活，无需预定义规则
- 能理解自然语言
- 能处理复杂场景

**劣势**：
- 需要调用大模型，增加延迟和成本
- 准确性依赖模型质量
- 需要更多测试验证

## 9. API 接口设计

### 9.1 工作流管理接口

**创建工作流**
- 功能：为任务创建新的工作流
- 参数：任务 ID、工作流定义
- 验证：检查工作流合法性
- 返回：工作流对象

**更新工作流**
- 功能：修改工作流定义
- 参数：工作流 ID、更新内容
- 限制：正在执行的工作流不可修改
- 返回：更新后的工作流对象

**删除工作流**
- 功能：删除工作流
- 参数：工作流 ID
- 限制：正在执行的工作流不可删除
- 返回：删除成功确认

**列出工作流**
- 功能：获取任务的所有工作流
- 参数：任务 ID
- 返回：工作流列表

### 9.2 执行控制接口

**启动执行**
- 功能：启动工作流执行
- 参数：工作流 ID、初始输入（可选）
- 返回：执行实例对象

**暂停执行**
- 功能：暂停正在执行的工作流
- 参数：执行 ID
- 返回：暂停成功确认

**恢复执行**
- 功能：恢复暂停的工作流
- 参数：执行 ID
- 返回：恢复成功确认

**取消执行**
- 功能：取消工作流执行
- 参数：执行 ID
- 返回：取消成功确认

**获取执行状态**
- 功能：查询执行状态
- 参数：执行 ID
- 返回：执行实例对象（包含当前状态、进度等）

### 9.3 日志查询接口

**获取执行日志**
- 功能：查询执行日志
- 参数：执行 ID、筛选条件（可选）
- 返回：日志列表

**导出日志**
- 功能：导出日志为文件
- 参数：执行 ID、格式（JSON/CSV）
- 返回：文件下载链接

## 11. 技术挑战与解决方案

### 11.1 异步消息等待

**挑战**：
- 发送消息后需要等待角色回复
- 回复时间不确定
- 可能超时或失败

**解决方案**：
- 使用异步运行时（Tokio）
- 设置超时机制
- 提供重试策略
- 支持手动干预

### 11.2 状态持久化

**挑战**：
- 执行过程中可能崩溃
- 需要恢复执行状态
- 保证数据一致性

**解决方案**：
- 每个节点执行后立即保存状态
- 使用事务保证原子性
- 提供状态恢复机制
- 定期检查点

### 11.3 并发控制

**挑战**：
- 多个工作流同时执行
- 资源竞争和死锁
- 性能瓶颈

**解决方案**：
- 使用消息队列缓冲任务
- 限制最大并发数
- 使用锁机制避免竞争
- 监控资源使用情况

### 11.4 循环检测

**挑战**：
- 工作流可能存在死循环
- 需要在运行前检测
- 需要在运行时保护

**解决方案**：
- 编辑器中进行拓扑排序检测
- 设置最大迭代次数
- 设置全局超时时间
- 提供循环检测工具

### 11.5 条件评估准确性

**挑战**：
- 关键词匹配可能不准确
- 正则表达式复杂难写
- AI 判断有延迟和成本

**解决方案**：
- 第一版使用简单规则
- 提供规则测试工具
- 逐步引入 AI 能力
- 允许用户手动干预

## 12. 风险评估与缓解

### 12.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 调度引擎性能瓶颈 | 高 | 中 | 使用消息队列、限制并发、性能测试 |
| 状态恢复失败 | 高 | 低 | 充分测试、提供手动恢复工具 |

## 15. 可行性结论

**技术可行性**: ✅ 高（但实施复杂）  
**实施复杂度**: 🔴 高  
**投入产出比**: 🟡 中等（取决于用户采用率）  
**建议**: 分阶段实施，先实现 MVP 版本验证价值

### 15.1 实施建议

1. **先实现 MVP 版本**（10 天）
   - 验证核心概念
   - 收集用户反馈
   - 评估实际使用场景

2. **根据反馈决定是否继续**
   - 如果用户采用率高，继续开发增强版本
   - 如果用户反馈一般，重新评估需求
   - 如果发现更重要的需求，调整优先级

3. **与任务模板系统协同**
   - 任务模板定义团队结构
   - 工作流定义协作流程
   - 两者结合提供完整解决方案

### 15.2 关键成功因素

- 用户界面简单易用
- 提供丰富的预置模板
- 执行稳定可靠
- 性能满足要求
- 文档和教程完善

该方案技术上可行，但实施复杂度较高。建议先完成任务模板系统，然后再启动智能调度系统的 MVP 版本开发。
