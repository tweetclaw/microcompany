# MicroCompany 需求文档

## 1. 一句话需求

把 Claude Code (Claurst) 的命令行对话界面改成图形界面。

## 2. 为什么要做这个

Claude Code 是一个强大的 AI 编程助手，但它是命令行工具。很多用户更习惯图形界面，特别是：

1. 对话历史更容易浏览和查找
2. 代码差异可以更直观地展示
3. 文件操作可以可视化
4. 不需要记忆命令行参数

## 3. 产品定义

MicroCompany 就是 Claude Code 的 GUI 版本。

**它是什么：**
- Claude Code 的图形界面包装
- 基于 Kuberwastaken 的 Claurst（Rust 实现）
- 保持 Claude Code 的所有能力
- 使用 Claude Code 的配置方式（settings.json）
- 复用 Claurst 的核心 crates

**它不是什么：**
- 不是新的 AI 框架
- 不是多智能体系统
- 不是工作流编排工具
- 不是项目管理工具

## 4. MVP 功能范围

### 4.1 必须做

1. **对话界面**
   - 文本输入框（替代命令行输入）
   - 对话历史显示（替代终端滚动）
   - 支持 Markdown 渲染
   - 代码高亮显示

2. **Claurst 集成**
   - 直接引用 Claurst 的 Rust crates
   - 调用 Claurst 的 API 发送消息
   - 接收并展示 AI 回复
   - 使用 Claurst 现有的配置文件

3. **基础功能**
   - 选择工作目录
   - 清空对话历史
   - 复制代码块
   - 保存对话记录

### 4.2 明确不做

第一版不做，后续版本升级需要实现的功能点：
- 多项目管理
- 角色配置界面
- 工作流画布
- 多 Agent 协作
- 自定义 prompt 编辑器
- 插件系统
- 团队协作功能

## 5. 技术约束

1. **必须使用 Claurst 作为后端**
   - 不重新实现 AI 对话逻辑
   - 不重新实现工具调用
   - 不重新实现模型配置

2. **必须兼容 Claurst 的配置**
   - 读取 `~/.claude/settings.json`
   - 使用用户已配置的模型
   - 使用用户已配置的 API keys

3. **必须保持简单**
   - 界面简洁
   - 功能聚焦
   - 易于维护

### 6.2 界面元素定义

**顶部工具栏：**
- 工作目录显示和选择按钮
- 清空对话按钮
- 设置按钮（后续版本）

**对话历史区域：**
- 用户消息：左对齐，浅色背景
- AI 回复：左对齐，深色背景
- Markdown 渲染支持
- 代码块语法高亮
- 每条消息的操作按钮（复制、重新生成）

**输入区域：**
- 多行文本输入框
- 发送按钮
- 停止生成按钮（生成时显示）

### 6.3 交互流程

1. **启动流程**
   ```
   用户打开应用 → 选择工作目录 → 进入对话界面
   ```

2. **对话流程**
   ```
   用户输入问题 → 点击发送 → 显示加载状态 → 流式显示 AI 回复 → 完成
   ```

3. **错误处理**
   ```
   API 错误 → 显示错误提示 → 允许重试
   网络错误 → 显示网络错误 → 允许重试
   配置错误 → 显示配置指引 → 跳转到设置
   ```

## 7. 成功标准

当以下条件满足时，MVP 成功：

1. 用户可以通过图形界面与 Claurst 对话
2. 对话体验不比命令行差
3. 所有 Claurst 的功能都可以使用
4. 用户不需要学习新的配置方式
5. 至少 3 个真实用户愿意从命令行切换到这个界面

## 8. 产品形态

- 桌面应用（Tauri）
- 单窗口界面
- 对话历史区域（占主要空间）
- 输入框和发送按钮（底部固定）
- 顶部：工作目录选择和基础设置

## 9. 开发原则

### 9.1 面向 UI 开发（UI First）

**原则：先确定产品的样子，用 UI 约束代码的接口和功能边界**

- 第一步：设计完整的 UI 界面和交互流程
- 第二步：根据 UI 需求定义前后端接口
- 第三步：实现接口和功能

**为什么这样做：**
- UI 是用户直接接触的部分，决定了产品的边界
- 先定义 UI 可以避免实现不必要的功能
- UI 驱动接口设计，确保接口符合实际需求

**具体做法：**
1. 画出所有界面的原型图
2. 定义所有用户交互流程
3. 根据 UI 需求列出需要的接口
4. 接口只实现 UI 需要的功能，不多不少

### 9.2 面向接口开发（Interface First）

**原则：开发完 UI 后，先定义接口，每个接口对应一个虚假实现**

- 第一步：根据 UI 需求定义所有接口
- 第二步：为每个接口创建 Mock 实现（返回假数据）
- 第三步：前端使用 Mock 接口完成 UI 开发
- 第四步：逐个替换 Mock 为真实实现

**为什么这样做：**
- 前后端可以并行开发
- 接口定义清晰，职责明确
- 可以先完成 UI，验证交互流程
- 降低前后端耦合

**具体做法：**
1. 定义 Tauri Command 接口
2. 创建 Mock 实现返回假数据
3. 前端基于 Mock 完成 UI
4. 后端实现真实逻辑
5. 替换 Mock 为真实实现

### 9.3 模块化开发（Module First）

**原则：所有模块独立并行，每个模块是一个具体功能接口的真实实现**

- 每个模块负责一个独立的功能
- 模块之间通过接口通信
- 模块可以独立开发、测试、替换

**为什么这样做：**
- 降低复杂度
- 提高可维护性
- 方便并行开发
- 易于测试和替换

**模块划分：**
1. **UI 模块**（React 组件）
   - ChatInterface: 对话界面
   - MessageList: 消息列表
   - InputBox: 输入框
   - Toolbar: 工具栏

2. **接口模块**（Tauri Commands）
   - init_session: 初始化会话
   - send_message: 发送消息
   - get_history: 获取历史
   - clear_history: 清空历史

3. **核心模块**（Rust 实现）
   - ClaurstWrapper: Claurst 封装
   - ConfigManager: 配置管理
   - SessionManager: 会话管理
   - MessageHandler: 消息处理
