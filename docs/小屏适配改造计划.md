# 系统 UI 小屏适配改造计划

## 1. 背景与目标

当前系统 UI 明显以大屏/桌面端为主要设计目标：
- 字体层级偏大屏阅读场景
- 左右面板长期常驻
- 任务列表、模板卡片、编辑区更偏并列布局
- 信息密度和留白比例更适合宽屏

这在普通对话界面和任务模板界面上都表现为相同问题：
当视口缩小时，主内容被侧栏、工具栏和检查区挤压，排版比例失衡，阅读与操作体验明显下降。

本次改造目标不是单独做一套“移动端 UI”，而是：

> 在保留当前产品视觉风格、交互体系和功能结构的前提下，为整套系统补齐一套统一的小屏适配策略，让普通对话、任务模式、模板选择、模板编辑等核心界面在小尺寸下依然清晰、可读、可操作。

---

## 2. 改造原则

### 2.1 不重做风格，只重构布局策略
保留现有：
- 主题与视觉语言
- 组件层级结构
- 普通模式 / 任务模式的核心交互路径

重点调整：
- 面板出现方式
- 栅格与列数
- 字号、间距、内容宽度
- 小屏优先级与滚动策略

### 2.2 主内容优先，次级面板按需展开
在小屏下：
- 主内容区必须优先展示
- 次级内容（侧栏、任务列表、检查区）应改为抽屉/覆盖层/折叠区
- 避免多个固定面板同时占据窄视口

### 2.3 先统一规则，再做页面改造
不建议逐页临时 patch。
应先统一：
- 断点
- token
- 面板行为
- 响应式约束

这样后续普通模式、任务模式、模板流才能共享一套小屏策略。

---

## 3. 现状判断

基于当前代码结构，项目已经具备一部分小屏基础，但尚未系统化。

### 3.1 已有可复用能力

#### `src/components/ChatInterface.tsx`
- 已存在 `window.innerWidth < 1280` 的 compact 模式逻辑
- 说明普通对话区已经有“小屏收缩”的实现基础

#### `src/components/Sidebar.tsx`
- 已支持 overlay / drawer 形式侧栏
- 适合作为系统统一的小屏导航模式

#### `src/components/InspectorPanel.tsx`
- 已支持右侧抽屉式检查面板
- 可作为任务细节/属性面板的小屏标准方案

#### `src/styles/variables.css`
- 已定义字体、间距、布局相关 token
- 已在 `1279px` 下收缩：
  - `--layout-sidebar-width`
  - `--layout-inspector-width`
  - `--layout-center-max-width`
  - `--layout-gutter`

### 3.2 当前主要问题

#### 1) 响应式能力集中在聊天主界面，未扩展到整套系统
普通聊天流已有 compact 基础，但任务模板流仍更接近桌面布局。

#### 2) 任务/模板区域仍保留明显桌面化结构
重点风险区域：
- `src/components/TaskListPanel.tsx`
- `src/components/TemplatePicker.tsx`
- `src/components/TemplateDraftEditor.tsx`

这些区域更可能存在：
- 固定宽度依赖
- 卡片最小宽度偏大
- 多栏并排结构
- 页头按钮拥挤
- 二级信息常驻占位

#### 3) 断点策略不够统一
目前：
- JS 中有 `1280px` compact 逻辑
- CSS 中有 `@media (max-width: 1279px)`
- 但缺少更明确的小屏 / 平板 / 桌面分层策略

#### 4) 一些布局尺寸仍然是组件内固定值
虽然全局已有布局 token，但任务相关区域仍可能继续依赖局部像素宽度，导致系统层的适配无法完全落地。

---

## 4. 总体改造方案

## Phase 1：统一响应式基础层

### 目标
先把系统的响应式“规则”定义清楚，而不是先改页面。

### 主要工作
以 `src/styles/variables.css` 为核心，补齐或标准化：
- 语义化断点
- 小屏间距缩放规则
- 面板宽度 token
- 内容最大宽度 token
- 不同尺寸下的字号密度策略

### 建议方向
当前已有：
- `--layout-sidebar-width`
- `--layout-inspector-width`
- `--layout-center-max-width`
- `--layout-gutter`

应进一步把以下内容纳入 token 控制：
- 小屏页边距
- 小屏标题/正文密度
- 抽屉宽度
- 卡片最小宽度
- 表单区块间距

### 预期收益
- 避免组件各自写死尺寸
- 让任务流与聊天流使用同一套响应式尺寸语言

---

## Phase 2：统一普通模式与任务模式的外层壳行为

### 目标
让 `NormalModeLayout` 和 `TaskModeLayout` 在小屏下遵循同一套布局原则。

### 重点文件
- `src/components/NormalModeLayout.tsx`
- `src/components/TaskModeLayout.tsx`
- `src/components/ChatInterface.tsx`
- `src/components/Sidebar.tsx`
- `src/components/InspectorPanel.tsx`

### 改造方向

#### 桌面端
- 保留多面板并列结构
- 保留侧栏与检查区常驻

#### 小屏端
统一降级规则：
- 左侧导航 → 抽屉
- 任务列表 → 抽屉或覆盖层
- 检查区 → 右侧抽屉
- 中央主工作区 → 始终优先保留

### 设计原则
- 窄视口下不同时保留多个固定侧栏
- 优先保持消息区 / 主编辑区完整可读
- 所有次级信息按需展开

### 预期收益
- 普通模式与任务模式不会在小屏上变成两套体验
- 降低维护成本

---

## Phase 3：重点改造任务/模板流

这是本次最关键的页面级改造。

### 5.1 `src/components/TaskListPanel.tsx`

#### 当前问题方向
- 更像桌面左侧固定面板
- 小屏时容易挤占主内容区

#### 改造目标
- 小屏下不作为固定列长期占位
- 优先转为抽屉式任务导航
- 保证任务切换功能可达，但不压缩主界面

---

### 5.2 `src/components/TemplatePicker.tsx`

#### 当前问题方向
- 模板卡片栅格更适合宽屏
- 卡片最小宽度过大时，小屏容易拥挤或出现非理想换行

#### 改造目标
- 小屏下卡片退化为单列
- 缩减卡片内信息密度
- 保证标题、说明、操作按钮可读可点

#### 原则
- 宽屏：保留现有卡片网格风格
- 小屏：优先单列、纵向滚动、弱化横向并排

---

### 5.3 `src/components/TemplateDraftEditor.tsx`

#### 当前问题方向
- 编辑区、角色卡片区、操作区更可能是多区域并排
- 小屏容易出现滚动冲突与信息压缩

#### 改造目标
- 在窄屏下改为纵向堆叠
- 次级信息区可折叠/延后展示
- 页头操作按钮允许换行或分层展示

#### 原则
- 用户当前编辑内容优先可见
- 操作按钮不能因为空间不足而变成“看见但点不到”

---

## 6. 统一交互规则

小屏适配不仅是样式问题，也包括交互一致性。

### 6.1 抽屉行为统一
需要统一：
- 打开入口位置
- 遮罩层表现
- 关闭方式（点击遮罩 / 返回 / 关闭按钮）
- 层级与 z-index
- 打开后页面滚动控制

### 6.2 页面内部滚动规则统一
避免：
- 抽屉可滚动
- 主内容也可滚动
- 卡片区再套一层滚动

目标是：
- 每个主要区域只保留必要滚动容器
- 减少嵌套滚动带来的触控体验问题

### 6.3 长文本策略统一
需要统一处理：
- 长标题
- 长路径
- 长模板名
- 长标签

避免在小屏下：
- 撑破布局
- 挤压按钮
- 导致横向滚动

### 6.4 点击区域与输入区域统一
重点验证：
- 按钮触控尺寸
- 输入框上下留白
- 页头图标按钮间距
- 抽屉中的列表项点击区域

## 8. 建议实施顺序

### Step 1
先审计所有固定宽度、多栏布局、最小卡片宽度、横向溢出风险。

### Step 2
先改 `variables.css`，统一断点与布局 token。

### Step 3
统一 `NormalModeLayout` / `TaskModeLayout` 的小屏壳逻辑。

### Step 4
处理任务列表、模板选择、模板编辑这三块最明显的桌面化残留。

### Step 5
最后处理滚动、抽屉、页头按钮、长文本等细节。

---

## 9. 风险点

- JS 与 CSS 的断点不一致，导致行为和样式错位
- 面板常驻/抽屉切换后状态变复杂
- 模板编辑页可能出现多层滚动冲突
- 页头操作按钮在小屏下易拥挤
- 如果任务模式继续单独演化，会与普通模式形成两套响应式逻辑

---

## 10. 验证方案

### 10.1 视口验证
必须至少验证：
- 375px
- 768px
- 1024px
- 1280px 附近
- 宽屏桌面

### 10.2 页面验证
逐页检查：
- 普通对话界面
- 任务模式主界面
- 模板选择界面
- 模板编辑界面

### 10.3 验证项
确认以下内容：
- 无横向溢出
- 主内容区始终优先、清晰、可操作
- 侧栏 / 任务列表 / 检查区可以正常打开关闭
- 卡片、表单、输入区在小屏仍可点击和阅读
- 抽屉打开后的滚动与遮罩逻辑正确
- 长文本不会破坏布局
- 普通模式与任务模式在小屏体验一致

---

## 11. 预期结果

完成后，系统应达到：
- 普通对话、任务模式、模板流共享同一套响应式原则
- 主内容优先，次级面板按需展开
- 大屏布局语言能自然收缩到小屏，而不是简单压缩
- 后续新增页面可以沿用同一套 token 与布局规则继续扩展
