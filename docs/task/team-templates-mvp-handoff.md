## 2. 问题定义

### 当前痛点
Task AI 模式下，用户每次创建多角色任务都需要重复完成以下配置：

- 手动添加角色
- 重复选择 archetype
- 重复配置 provider/model
- 重新搭建相似团队结构

这导致：

1. 有效团队结构无法沉淀
2. 多角色协作经验无法复用
3. 创建 task 的启动成本偏高
4. 团队协作入口不够标准化

### 本功能要解决什么
团队模板功能要解决的是：

> 让用户能把“已有的一套可复用团队配置”保存下来，并在下次创建 task 时快速复用，同时不破坏现有 Task/Role/Session 运行时约束。

## 3. MVP 范围结论

## 3.1 最小可用功能集

建议将 MVP 严格收敛为以下 4 项：

### A. 从系统模板创建 Task
用户可以在新建 task 时选择一个系统模板作为起点。

**必须包含**：
- 模板列表
- 模板基本预览（名称、描述、角色列表、推荐 archetype）
- 选择模板后生成 task draft

**不必在 MVP 包含**：
- 高级筛选/搜索
- 模板标签体系
- 模板评分、推荐、市场

### B. 创建前确认与补全配置
模板不能直接变成运行时 task，必须先进入创建前确认。

**必须包含**：
- task draft 可编辑
- 可修改 task 名称
- 可修改 role 名称
- 可修改 archetype
- 可补齐 provider/model
- provider/model 未补齐时禁止创建

**这是 MVP 的关键闭环，不可省略。**

### C. 保存当前 Task 为用户模板
用户在已有 task 中调好团队结构后，可以保存成自己的模板。

**必须包含**：
- “保存为模板”入口
- 输入模板名（必填）
- 可选描述
- 读取当前 task roster 并保存为用户模板

**可以延后**：
- 编辑已有用户模板
- 复杂模板管理页
- 模板复制、版本历史、导入导出

### D. 用户模板的读取与再次使用
用户保存后的模板，下次创建 task 时可以再次选中并复用。

**必须包含**：
- 用户模板列表
- 与系统模板并列展示或分组展示
- 从用户模板进入同一套“创建前确认”流程

---

## 3.2 明确排除项（避免范围失控）

以下内容不建议进入本次 MVP：

1. **动态添加/删除/更新团队成员**
   - 虽然 `team-templates-user-guide.md` 将其列入完整能力，但它本质上是“运行时团队管理”，不是“模板 MVP 必需闭环”的一部分。
   - 若一起做，会把范围从模板能力扩大到 task roster management。

2. **用户模板编辑/删除完整后台**
   - MVP 只要能保存并再次使用即可。
   - 编辑、删除可以作为 next step。

3. **模板协作工作流/自动 handoff 路径**
   - 当前阶段模板只负责复用 roster，不负责自动化流程编排。

4. **模板市场 / 分享 / 同步**
   - 远期功能，完全排除。

5. **模板与已创建 task 的双向绑定**
   - 明确禁止。模板是创建来源，不是运行时绑定对象。

---

## 4. 关键用户流程

## 4.1 流程一：从模板创建 Task（核心主流程）

### 用户目标
快速创建一个标准团队，而不是从空白逐个加人。

### 目标流程
1. 用户点击 `New Task`
2. 选择“从模板创建”
3. 浏览系统模板 / 用户模板
4. 查看模板预览
5. 选择模板后，系统生成 `TaskTemplateDraft`
6. 进入“创建前确认”页面
7. 用户补全或修改：
   - task 名称
   - role 名称
   - archetype
   - provider/model
8. 系统校验所有 role 都具备完整 provider/model
9. 调用现有 `create_task` 流程创建 task / roles / sessions
10. 进入正常 Team Brief + handoff 运行时流程

### 关键产品约束
- 模板可以不完整，但 task 不可以
- 创建前确认是必经步骤，不是优化项
- 模板创建后的 task 与模板脱钩

---

## 4.2 流程二：保存当前 Task 为模板（核心沉淀流程）

### 用户目标
把一套已经验证有效的团队配置沉淀成可复用资产。

### 目标流程
1. 用户在某个 task 中完成团队配置
2. 点击“保存为模板”
3. 输入模板名称、可选描述
4. 系统读取当前 task 的角色结构：
   - 角色名称
   - 身份描述
   - archetype
   - provider/model
   - display order
5. 保存为用户模板
6. 下次创建 task 时出现在用户模板列表

### 关键产品约束
- 只保存团队结构，不保存会话历史
- 保存后与原 task 脱钩
- 允许用户从同一个 task 多次保存不同模板

---

## 4.3 流程三：使用用户模板再次创建 Task

### 用户目标
真正复用自己沉淀下来的团队配置。

### 目标流程
1. 用户点击 `New Task`
2. 选择“从模板创建”
3. 在用户模板分组里选择自己的模板
4. 进入创建前确认页面
5. 进行必要检查和修改
6. 创建 task

### 关键意义
如果没有这条流程，“保存为模板”就没有业务闭环，因此必须纳入 MVP。

---

## 5. 技术实现要点

## 5.1 数据模型边界要清晰

必须严格区分 4 层对象：

1. `TaskTemplate`
   - 可复用模板资产
2. `TemplateRole`
   - 模板中的角色定义
3. `TaskTemplateDraft`
   - 模板复制后的前端临时可编辑态
4. `TaskCreateRequest` / `Task` / `TaskRole`
   - 真正运行时对象

### 关键原则
**不要让模板对象直接承担运行时对象职责。**

否则会导致：
- provider/model 空值和运行时非空约束冲突
- 模板修改污染已创建 task
- 创建流程复杂度上升

---

## 5.2 模板到 Draft，再到 Task

实现路径建议固定为：

`TaskTemplate` → `TaskTemplateDraft` → `create_task`

### 为什么这样做
因为模板允许 provider/model 为空，而现有 task 创建链路要求完整配置。

### 实现收益
- 复用现有 `create_task` 逻辑
- 不改动运行时 schema 基本假设
- 把模板能力限制在创建前层面，风险更低

---

## 5.3 存储策略

### 系统模板
建议按文档既定方案：

- 资源目录：`src-tauri/resources/task-templates/`
- 应用启动时同步到本地目录
- 运行时统一从本地目录读取

### 用户模板
建议存储在应用数据目录，由后端命令统一 CRUD。

### 关键原因
系统模板和用户模板的覆盖策略不同：
- 系统模板可随版本升级覆盖
- 用户模板属于用户资产，不能被覆盖

---

## 5.4 最小后端接口建议

MVP 至少需要这些能力：

1. `list_task_templates`
   - 返回系统模板 + 用户模板（可分组）

2. `get_task_template`
   - 返回模板详情，用于预览与 draft 生成

3. `save_task_as_template`
   - 从当前 task 提取结构并保存为用户模板

4. `create_task_from_draft` 不一定必须新开接口
   - 更推荐前端把 draft 转成现有 `create_task` 请求，直接复用原命令

### 可延后接口
- update user template
- delete user template
- duplicate template

---

## 5.5 前端最小界面建议

MVP 推荐拆成 3 个最小界面块：

### 1. 新建任务入口扩展
- 空白创建
- 从模板创建

### 2. 模板列表 + 模板预览
- 系统模板/用户模板分组
- 显示模板名、描述、角色数、角色列表

### 3. 创建前确认页
- draft 编辑
- 缺失 provider/model 高亮
- 创建按钮禁用/拦截

### 4. Task 页中的“保存为模板”入口
- 简单弹窗即可
- 不需要复杂管理台

---

## 5.6 校验与错误处理

MVP 必须覆盖以下校验：

1. 模板 role 缺少 provider/model
   - 允许进入 draft
   - 不允许直接创建 task

2. role 名称为空或重复
   - 创建前阻止

3. archetype 不存在或资源缺失
   - 给出明确错误
   - 允许用户重新选择 archetype

4. 系统模板文件损坏/格式错误
   - 跳过该模板并记录错误，不应阻断整个模板列表

5. 用户模板读取失败
   - 不影响系统模板加载

---

## 5.7 与现有能力的衔接点

这个功能不是独立系统，必须明确复用现有能力：

- **Archetype**：模板 role 通过 `recommendedArchetypeId` 关联 archetype
- **Prompt Snapshot**：task 创建后仍按既有逻辑生成
- **Team Brief**：task 创建成功后照常展示团队结构
- **Handoff**：沿用当前协议，不因模板而改变

### 产品原则
团队模板只解决：
> 如何更快创建正确的团队

它**不负责**改变运行时协作协议。

## 6. 推荐的 MVP 交付切分

建议按以下顺序推进，方便 bob / clark 并行：

### Slice 1：后端模板读取能力
- 定义 `TaskTemplate` / `TemplateRole`
- 加载系统模板
- 读取用户模板
- 提供列表/详情命令

### Slice 2：前端模板选择与预览
- 新建 task 增加模板入口
- 模板列表
- 模板预览

### Slice 3：Draft 与创建前确认
- 模板复制为 draft
- 编辑 role 配置
- 校验 provider/model
- 复用现有 task 创建命令

### Slice 4：保存 task 为用户模板
- Task 页面增加保存入口
- 后端保存用户模板
- 模板列表展示用户模板

## 7. 风险与取舍

## 7.1 最大风险
把“团队模板”与“运行时团队动态管理”一起做，导致范围失控。

### 建议取舍
本轮先只交付：
- 模板选用
- 创建前确认
- 保存为模板
- 再次复用

把以下内容后移：
- 动态增删角色
- 用户模板管理台
- 模板编辑/删除

---

## 7.2 第二个风险
试图让模板直接创建 session，跳过 draft。

### 为什么危险
这会直接撞上 provider/model 非空约束，导致需要大改现有 runtime schema 或 create_task 流程。

### 建议
坚持 draft 中转，不绕开现有链路。

## 9. 建议的验收标准（MVP版）

本次 MVP 建议以以下 5 条作为完成标准：

1. 用户能查看系统模板和用户模板列表
2. 用户能从模板生成 task draft
3. 用户能在创建前补全并修改 role 配置
4. 用户能把当前 task 保存为用户模板
5. 用户能再次使用已保存的用户模板创建 task

如果上述 5 条达成，就已经形成完整的 MVP 闭环。

动态团队成员管理可作为下一阶段能力，而不是当前发布阻塞项。
